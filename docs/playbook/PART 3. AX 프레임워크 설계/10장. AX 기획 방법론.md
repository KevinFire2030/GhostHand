## 소목차

## 디자인 씽킹

AX를 설계할 때 가장 먼저 버려야 할 오해 중 하나는, 에이전트 시스템이 단지 기술 조합의 문제라는 생각이다.  
물론 모델 선택, 도구 연결, 인터페이스 구현은 모두 중요하다. 그러나 사용자가 실제로 체감하는 가치는 기술 그 자체가 아니라, 그 기술이 어떤 문제를 얼마나 자연스럽게 해결하느냐에서 나온다. 그래서 AX 설계의 출발점은 기술이 아니라 사용자와 맥락이어야 한다.

이 지점에서 유효한 접근이 디자인 씽킹이다.  
디자인 씽킹은 정답을 먼저 정해놓고 기능을 끼워 맞추는 방식이 아니라, 사용자의 문제를 깊이 이해하고, 그 문제를 다양한 관점에서 재정의하고, 해결책을 반복적으로 검증해 가는 접근이다. 모바일 AI 에이전트처럼 아직 익숙하지 않은 경험을 설계할 때일수록 이 접근이 중요해진다.

왜냐하면 에이전트는 단순한 기능 개선이 아니라, 사용자의 행동 방식 자체를 바꾸기 때문이다.  
기존 앱은 사용자가 버튼을 누르고 기능을 호출하는 흐름에 맞춰 설계되었다. 하지만 에이전트는 사용자가 목표를 말하고, 시스템이 이를 해석하고, 필요한 작업을 대신 수행하는 방식으로 흐름을 바꾼다. 이런 변화는 작은 UI 개선 수준이 아니라, 서비스 사용 방식 전체를 재구성하는 일에 가깝다.

디자인 씽킹은 이런 전환을 다룰 때 특히 유용하다.  
사용자가 어디에서 불편을 느끼는지, 어떤 반복 작업에 시간을 빼앗기는지, 어떤 순간에 도움받고 싶어 하는지, 무엇을 불안해하는지를 먼저 이해하게 해주기 때문이다. 예를 들어 사용자가 “아침마다 뉴스를 확인하고 일정을 정리하고 중요한 메일을 확인하는 것이 번거롭다”는 문제를 갖고 있다면, 에이전트는 단순히 뉴스 요약 기능 하나가 아니라 “하루 시작 루틴을 자동화하는 경험”으로 재정의될 수 있다.

즉 디자인 씽킹은 기능을 발명하는 방법이 아니라, 문제를 인간 중심으로 다시 보는 방법이다.  
AX에서는 이 점이 특히 중요하다. 에이전트는 사람 대신 행동할 수 있는 만큼, 잘못 설계하면 편리함보다 불안과 거부감을 키울 수도 있다. 따라서 단순히 “무엇을 자동화할 수 있는가”보다 “사용자가 어디까지 위임하고 싶어 하는가”, “어떤 결과를 신뢰할 수 있다고 느끼는가”, “어떤 순간에 개입하고 싶어 하는가”를 먼저 이해해야 한다.

결국 디자인 씽킹은 AX 기획의 감성적 장식이 아니라, 에이전트 설계의 출발점이다.  
기술은 수단이지만, 경험은 목적이기 때문이다. 그리고 AX가 진짜 유용한 경험이 되려면, 먼저 인간의 문제를 정확히 읽어낼 수 있어야 한다.

## 5D 프로세스 (Define–Discover–Design–Develop–Deploy)

AX를 실무적으로 기획하려면 추상적인 감각만으로는 부족하다.  
사용자의 문제를 이해하는 것과, 그것을 실제 서비스 구조로 전환하는 것 사이에는 체계적인 프로세스가 필요하다. 여기서 유용하게 적용할 수 있는 방식이 5D 프로세스다. 이 책에서는 모바일 AI 에이전트와 AX 서비스를 기획하기 위한 실전적 틀로 **Define–Discover–Design–Develop–Deploy**의 5단계를 제안한다.

### 1) Define
첫 번째 단계는 Define, 즉 문제를 정의하는 단계다.  
이 단계에서 중요한 것은 “무엇을 만들까”보다 “어떤 문제를 해결하려는가”를 분명히 하는 것이다. 예를 들어 “메신저 기반 AI 비서 만들기”는 솔루션이지 문제 정의가 아니다. 진짜 문제 정의는 “사용자가 아침마다 여러 앱을 오가며 브리핑을 준비하는 피로를 줄여야 한다”처럼 구체적이어야 한다.

### 2) Discover
두 번째 단계는 Discover다.  
이 단계에서는 사용자의 실제 맥락과 제약을 탐색한다. 사용자는 어떤 루틴을 반복하는가, 어떤 지점에서 앱 전환 피로를 느끼는가, 무엇을 자동화하고 싶어 하는가, 무엇은 직접 통제하고 싶어 하는가를 관찰하고 정리한다. AX에서는 특히 “위임 가능한 영역”과 “사용자가 반드시 직접 확인하고 싶은 영역”을 구분하는 것이 중요하다.

### 3) Design
세 번째 단계는 Design이다.  
여기서는 사용자 문제를 에이전트 흐름으로 바꾼다. 어떤 인터페이스를 사용할지, 메신저와 Web UI의 역할을 어떻게 나눌지, 어떤 스킬 조합으로 요청을 처리할지, 어디에 확인 단계와 예외 처리를 둘지를 설계한다. 이 단계는 화면 설계뿐 아니라, 에이전트의 역할 분담과 행동 규칙을 함께 설계하는 단계다.

### 4) Develop
네 번째 단계는 Develop이다.  
이제 설계한 흐름을 실제로 구현한다. OpenClaw 기반 플랫폼, RAG 구조, Web UI, 메신저 인터페이스, 스마트폰 제어 도구, 자동화 흐름을 연결한다. 이때 중요한 것은 처음부터 완벽한 시스템을 만들기보다, 가장 핵심 시나리오를 빠르게 동작시키는 것이다. AX는 종종 실제 써봐야만 드러나는 문제가 많기 때문이다.

### 5) Deploy
마지막 단계는 Deploy다.  
배포는 단순히 실행 환경에 올리는 것이 아니라, 실제 사용자 경험 안으로 에이전트를 투입하는 단계다. 이때는 단순 가동 여부보다, 사용자가 얼마나 자연스럽게 받아들이는지, 어디에서 이탈하는지, 어떤 부분에서 신뢰를 얻거나 잃는지를 함께 관찰해야 한다. AX는 배포 이후에야 비로소 진짜 검증이 시작된다.

5D 프로세스의 장점은 단순한 개발 절차가 아니라, 에이전트 경험을 단계적으로 검증하게 만든다는 점이다.  
특히 모바일 AI 에이전트처럼 사용자 행동 자체를 바꾸는 서비스에서는, 개발보다도 정의와 발견, 그리고 배포 이후의 관찰이 훨씬 중요하다. 5D는 바로 이 점을 구조적으로 강제해주는 틀이다.

## AX 서비스 기획

AX 서비스 기획은 일반적인 앱 기획과 결이 다르다.  
기존 앱 기획이 보통 화면 구조, 기능 목록, 사용자 흐름, 클릭 경로 중심으로 진행되었다면, AX 서비스 기획은 “무엇을 자동으로 맡길 것인가”, “어디까지 에이전트가 판단하게 할 것인가”, “어디에서 사람이 개입해야 하는가” 같은 질문을 중심으로 전개된다.

즉 AX 서비스는 기능을 나열하는 방식으로 기획하기보다, **사용자 목표와 위임 흐름**을 중심으로 기획해야 한다.  
예를 들어 기존 기획 문서에서는 “뉴스 화면”, “일정 화면”, “알림 화면”처럼 화면 단위로 정리할 수 있지만, AX 서비스에서는 “아침 브리핑”, “회의 준비”, “생활 자동화”, “문서 생산 자동화”처럼 목표 단위로 구조화하는 편이 더 자연스럽다.

AX 서비스 기획에서 중요한 관점은 세 가지다.

첫째, **목표 중심성**이다.  
사용자는 앱 기능을 쓰기 위해 서비스를 찾는 것이 아니라, 어떤 결과를 얻기 위해 서비스를 찾는다. 따라서 “일정 보기”보다 “내일 하루 준비”, “문서 요약”보다 “회의 전 핵심 파악”처럼 목표를 중심으로 기획해야 한다.

둘째, **위임 설계**다.  
사용자가 어디까지 AI에게 맡기고 싶어 하는지를 정교하게 설계해야 한다. 모든 것을 자동화하는 것이 늘 정답은 아니다. 오히려 적절한 지점에서 사용자의 승인, 수정, 최종 검토를 넣어야 신뢰가 생긴다. AX 서비스는 자동화 비율이 아니라 위임의 질이 중요하다.

셋째, **실행 가능성**이다.  
AI가 멋진 답을 내놓는 것과, 실제로 스마트폰에서 행동을 수행하는 것은 다른 문제다. 따라서 서비스 기획 단계에서부터 어떤 도구가 있는지, 어떤 제어가 가능한지, 어떤 제약이 있는지를 고려해야 한다. AX 서비스는 이상적인 시나리오만으로 설계하면 안 되고, 실제 실행 가능한 범위를 중심으로 성장해야 한다.

AX 서비스 기획은 결과적으로 하나의 경험을 설계하는 일이다.  
사용자가 명령을 내리고, 에이전트가 이해하고, 적절히 판단하고, 도구를 호출하고, 중간 상태를 보여주고, 결과를 전달하며, 필요 시 인간의 개입을 받는 일련의 흐름 전체를 설계하는 것이다. 즉 화면 하나가 아니라, 사람과 에이전트의 협업 프로토콜을 설계하는 일에 가깝다.

모바일 AI 에이전트 시대에는 기능보다 흐름이 중요해진다.  
그리고 그 흐름을 잘 설계하려면, 사용자 문제를 이해하는 디자인 씽킹, 실행 중심의 5D 프로세스, 위임과 신뢰를 고려한 AX 서비스 기획이 함께 필요하다.

## 정리

AX 프레임워크 설계의 출발점은 기술이 아니라 사용자 문제다.  
디자인 씽킹은 그 문제를 인간 중심으로 다시 보게 만들고, 5D 프로세스는 그것을 실행 가능한 구조로 전환하게 해준다. 그리고 AX 서비스 기획은 단순한 기능 나열이 아니라, 사람과 에이전트의 협업 흐름 자체를 설계하게 만든다.

결국 좋은 AX 서비스는 “AI가 대신 해주는 기능”의 집합이 아니다.  
그것은 사용자의 목표를 이해하고, 적절히 위임을 설계하고, 실행 가능한 구조로 구현하고, 실제 경험 속에서 조정되는 살아 있는 시스템이다. 바로 이 점에서 AX 기획은 기존 앱 기획과 본질적으로 다른 차원을 갖는다.
