# C-4. 에이전트 프로젝트를 발명 문서로 바꾸는 법

에이전트 프로젝트를 하다 보면 어느 순간 이런 생각이 든다.  
“이거 그냥 구현으로 끝내기엔 아깝지 않나?”

실제로 많은 좋은 프로젝트가 그 단계에서 멈춘다.  
코드는 남아 있고, 데모는 있고, 메모도 있고, 심지어 사람들의 반응도 좋다. 그런데 그것이 제안서나 직무발명 신고서, 특허 아이디어, 제도 문서 같은 더 오래 남는 자산으로는 이어지지 못한다. 이유는 단순하다. 구현과 문서화는 다른 작업이기 때문이다. 구현은 동작을 만드는 일이고, 발명 문서화는 그 동작의 구조와 차별점을 **제도적 언어로 번역하는 일**이다.

그래서 중요한 것은 “좋은 프로젝트를 했는가”만이 아니라, “그 프로젝트를 자산 언어로 다시 쓸 수 있는가”다.  
이 장에서는 에이전트 프로젝트를 발명 문서로 전환할 때 실제로 어떤 순서와 관점으로 접근하면 좋은지를 정리한다. 핵심은 거창한 법률 문서를 갑자기 쓰려 하지 말고, 프로젝트 안에 이미 들어 있는 문제, 차별점, 구조, 효과를 하나씩 끌어내는 것이다.

---

## 1. 프로젝트 메모부터 모아라

발명 문서화의 출발점은 완성된 보고서가 아니라, 보통 흩어진 메모다.  
아이디어를 적어둔 노트, 회의 메모, 기능 설명, 테스트 기록, 구현 중 남긴 코멘트, README, 브레인스토밍 자료 같은 것들이 전부 재료가 된다. 많은 사람들이 발명 문서를 쓰려 할 때 처음부터 완성된 문장으로 쓰려다 막히는데, 실제로는 먼저 프로젝트의 흔적을 모으는 쪽이 훨씬 중요하다.

특히 에이전트 프로젝트는 구조가 복합적이라 메모가 여러 곳에 흩어져 있을 가능성이 높다.  
예를 들어 다음 자료를 먼저 모아보면 좋다.

- 프로젝트 개요 메모
- README와 사용 설명서
- 기능 구조도
- 테스트 시나리오
- 데모 설명 자료
- 회의에서 나온 차별점 정리
- 구현 중 고민했던 우회 경로나 실패 처리 방식

이 단계의 핵심은 문장을 잘 쓰는 것이 아니다.  
오히려 “무엇이 이미 프로젝트 안에 들어 있었는가”를 발견하는 것이다. 좋은 발명 문서의 많은 재료는 이미 프로젝트 안에 들어 있는데, 다만 아직 제도 문서 형식으로 정리되지 않았을 뿐이다.

---

## 2. 문제 정의를 먼저 다시 써라

발명 문서화에서 가장 먼저 강하게 잡아야 하는 것은 문제 정의다.  
사람들은 흔히 자기가 만든 기능을 설명하는 데서 시작하려 하지만, 좋은 발명 문서는 “무엇을 만들었는가”보다 “기존 무엇이 부족했는가”에서 시작된다. 즉 프로젝트를 다시 보면, 기능 목록보다 먼저 문제를 써야 한다.

이때 도움이 되는 질문은 다음과 같다.

- 기존 방식은 왜 불편했는가?
- 어떤 반복이 비효율적이었는가?
- 무엇이 사람 손에 너무 많이 의존했는가?
- 기존 자동화 구조의 한계는 무엇이었는가?
- 왜 기존 도구만으로는 충분하지 않았는가?

예를 들어 모바일 AI 에이전트라면 단순히 “자연어로 제어했다”보다,
- 외부 PC 의존이 컸다
- 동일 단말 내부 제어 구조가 부족했다
- 실행 결과 검증과 재시도가 약했다
- 복수 실행 경로를 선택하는 구조가 없었다

처럼 구조적 문제를 먼저 정리하는 쪽이 훨씬 강하다.

즉 발명 문서화는 결과 자랑이 아니라, **문제를 구조적으로 다시 쓰는 일**에서 시작한다.

---

## 3. 기술 차별점을 기능이 아니라 구조로 뽑아라

초심자가 가장 자주 하는 실수는 차별점을 기능 목록처럼 쓰는 것이다.  
예를 들어 “요약 가능”, “자동화 가능”, “스마트폰 제어 가능”처럼 쓰면 얼핏 좋아 보이지만, 발명 문서 관점에서는 힘이 약할 수 있다. 왜냐하면 이런 문장은 너무 넓고, 이미 어디서나 볼 수 있는 기능처럼 보일 위험이 크기 때문이다.

대신 차별점은 기능이 아니라 구조로 뽑아야 한다.  
예를 들어 이렇게 바꾸는 식이다.

- 단순 자연어 처리 → 자연어 해석 + 실행 파라미터 추출 구조
- 단순 자동 실행 → 검증 기반 폐루프 재실행 구조
- 단순 모바일 제어 → 동일 단말 내부 자기 단말 제어 구조
- 단순 명령 실행 → 정책 판단 + 실행 경로 선택 구조

즉 차별점은 “무엇을 했다”보다 “어떤 방식으로 결합했고 어떤 구조를 만들었는가”로 표현될수록 발명 문서에 가까워진다.  
프로젝트를 자산화할 때 가장 중요한 감각이 바로 이 구조적 차별화 감각이다.

---

## 4. 핵심 구성도를 그려라

프로젝트가 복잡할수록 글만으로는 오히려 구조가 잘 안 보일 수 있다.  
그래서 발명 문서화에서는 핵심 구성도나 흐름도를 한 번 그려보는 것이 매우 중요하다. 그림이 꼭 정교할 필요는 없다. 중요한 것은 프로젝트를 “무엇과 무엇이 연결된 구조인가”로 한 번에 보이게 만드는 것이다.

예를 들어 에이전트 프로젝트라면 다음처럼 그려볼 수 있다.
- 입력
- 자연어 해석
- 정책 판단
- 실행 경로 선택
- 제어 실행
- 결과 검증
- 재시도

혹은
- 사용자
- 에이전트
- 문서 기억
- 도구/브리지
- 실행 대상 단말
- 검증 결과

이런 흐름도를 그려보면, 프로젝트를 단순 데모가 아니라 시스템 구조로 다시 볼 수 있게 된다.  
발명 문서는 결국 구조를 설명하는 문서이기 때문에, 구성도를 그리는 순간 생각이 훨씬 정리된다.

---

## 5. 선행기술과 비교할 관점을 먼저 정리하라

발명 문서화에서 중요한 또 하나의 포인트는 선행기술과의 비교다.  
여기서 많은 사람이 겁을 먹는다. 모든 논문과 특허를 다 읽어야 할 것 같고, 비교 작업이 너무 거대하게 느껴지기 때문이다. 하지만 처음에는 그렇게까지 갈 필요는 없다. 먼저 비교 관점부터 정리하면 된다.

즉 “무엇과 비교할 것인가?”보다 “어떤 축으로 비교할 것인가?”가 먼저다.  
예를 들어 다음 같은 비교 축을 만들 수 있다.

- 제어 주체: 외부 시스템인가, 동일 단말 내부 AI인가
- 제어 대상: 외부 단말인가, 자기 단말인가
- 입력 방식: 스크립트/규칙인가, 자연어 기반 의도 해석인가
- 실행 구조: 단일 경로인가, 복수 경로 선택인가
- 실행 이후: 명령 전달인가, 결과 검증 포함인가
- 실패 대응: 수동 재시도인가, 대체 경로 자동 재시도인가

이런 비교 축만 정리해도, 나중에 실제 선행기술을 붙일 때 훨씬 덜 막힌다.  
발명 문서화의 핵심은 선행기술을 무작정 많이 읽는 것이 아니라, **내 프로젝트를 비교 가능한 구조로 만드는 것**이다.

---

## 6. 기대 효과를 사용자 편의가 아니라 기술 효과로 써라

에이전트 프로젝트를 설명할 때는 종종 “편하다”, “쉽다”, “빠르다” 같은 표현을 많이 쓰게 된다.  
물론 그것도 중요하다. 하지만 발명 문서에서는 조금 더 기술적이고 구조적인 효과로 정리하는 편이 강하다.

예를 들어 다음처럼 바꿔 쓸 수 있다.

- 편리하다 → 자연어 입력만으로 복잡한 제어를 수행할 수 있다
- 빠르다 → 반복 작업의 수행 단계를 단축할 수 있다
- 잘 된다 → 검증 기반 구조로 신뢰성을 높일 수 있다
- 안전하다 → 민감 작업에 대해 정책 기반 확인 절차를 둘 수 있다
- 활용 범위가 넓다 → 복수 경로 선택과 복합 명령 처리로 응용 범위를 확장할 수 있다

즉 기대 효과는 감탄사가 아니라, **구조가 만들어내는 결과**로 적는 것이 좋다.  
이렇게 해야 프로젝트 설명이 발명 문서 언어로 더 자연스럽게 올라간다.

---

## 7. 마지막에는 청구항 감각으로 정리해보라

반드시 특허 명세서 수준으로 바로 갈 필요는 없지만, 마지막에는 한 번쯤 “이 프로젝트의 핵심 권리 범위는 어디까지일까?”를 생각해보는 것이 좋다.  
이게 바로 청구항 감각이다. 즉 핵심적인 구조를 문장 하나로 추상화해보는 연습이다.

예를 들어 이런 식이다.
- 자연어 명령을 해석하고
- 단말 동일성을 판단하고
- 정책 판단 결과를 반영해 실행 경로를 선택하고
- 실행 결과를 검증하며
- 실패 시 대체 경로로 재실행하는 장치 및 방법

이런 식으로 한 번 묶어보면, 프로젝트의 진짜 핵심이 어디까지인지 스스로도 더 선명하게 보인다.  
청구항 감각은 법률 작업 이전에, 프로젝트의 핵심 구조를 잘라내는 훈련이기도 하다.

---

## 결국 발명 문서화는 프로젝트를 다시 이해하는 일이다

에이전트 프로젝트를 발명 문서로 바꾼다는 것은 단순히 형식을 바꾸는 일이 아니다.  
그것은 프로젝트를 다시 읽고, 다시 구조화하고, 다시 차별점을 뽑아내고, 다시 자산의 언어로 번역하는 일이다. 그래서 이 과정은 힘들지만 매우 가치 있다. 구현할 때는 보이지 않던 구조가 더 또렷해지고, 프로젝트가 무엇을 남길 수 있는지 더 선명해지기 때문이다.

이 장의 흐름을 다시 정리하면 이렇다.

1. 프로젝트 메모를 모은다  
2. 문제 정의를 다시 쓴다  
3. 차별점을 구조로 뽑는다  
4. 핵심 구성도를 그린다  
5. 선행기술 비교 축을 만든다  
6. 기대 효과를 기술 효과로 정리한다  
7. 청구항 감각으로 핵심 구조를 묶는다

이 순서를 따라가면, 구현으로 끝날 뻔했던 프로젝트도 훨씬 더 오래 남는 자산으로 바뀔 수 있다.  
결국 발명 문서화란 결과를 포장하는 일이 아니라, **프로젝트 안에 이미 들어 있던 자산성을 발견하고 구조화하는 일**이다. 그리고 바로 그 지점에서, 에이전트 프로젝트는 단순한 실험을 넘어 조직 안에서 남을 수 있는 발명으로 변하기 시작한다.
