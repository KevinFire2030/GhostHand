## 모바일 에이전트 구조

모바일 AI 에이전트를 제대로 이해하려면 먼저 ‘무엇이 하나의 시스템을 이루는가’를 구조적으로 볼 필요가 있다.  
많은 사람들은 에이전트를 단순히 대화형 AI 하나로 생각하지만, 실제로 동작하는 모바일 에이전트는 훨씬 더 많은 계층으로 구성된다. 말하자면 에이전트는 모델 하나가 아니라, 여러 층위가 결합된 실행 시스템이다.

가장 바깥에서 사용자가 만나는 층은 인터페이스다.  
메신저, 웹 UI, 음성 명령, 혹은 간단한 버튼 기반 대시보드가 여기에 해당한다. 사용자는 이 인터페이스를 통해 에이전트에게 목표를 전달한다. “회의 내용을 정리해줘”, “내일 아침 브리핑 준비해줘”, “스마트폰 설정을 바꿔줘” 같은 요청은 모두 이 입구에서 시작된다.

그 다음 층은 오케스트레이션 레이어다.  
이 층은 사용자의 요청을 해석하고, 어떤 도구를 쓸지, 어떤 순서로 작업을 진행할지, 어느 지점에서 추가 확인이 필요한지를 결정한다. 쉽게 말해 에이전트의 ‘작업 관리자’에 해당하는 층이다. 대화 모델이 단순한 문장 생성에 머문다면, 오케스트레이션 레이어는 그 문장을 실제 실행 흐름으로 바꾸는 역할을 맡는다.

그 아래에는 모델 레이어가 있다.  
여기에는 언어 모델, 요약 모델, 음성 처리 모델, 검색 보조 모델 등 다양한 AI 모델이 들어갈 수 있다. 중요한 점은 모바일 에이전트에서 모델은 전체의 일부일 뿐이라는 것이다. 모델은 사고를 담당할 수는 있지만, 실제 행동은 다른 계층과 연결되어야만 완성된다.

다음은 도구와 스킬 레이어다.  
도구는 파일 읽기, 웹 검색, 일정 확인, 알림 발송, 스마트폰 제어, API 호출 같은 개별 행동 단위다. 스킬은 이러한 도구들을 특정 목적에 맞게 조합한 재사용 가능한 작업 패턴에 가깝다. 예를 들어 “아침 브리핑 생성”이라는 스킬은 뉴스 수집, 일정 조회, 우선순위 정리, 메신저 전달 같은 여러 도구를 묶어놓은 형태가 될 수 있다.

그 아래에는 메모리와 지식 레이어가 있다.  
에이전트가 매번 새로 시작하는 것이 아니라, 사용자의 선호, 과거 작업 이력, 문서, 노트, 저장된 지식베이스를 참고할 수 있게 만드는 층이다. 여기에 RAG나 개인 문서 검색 구조가 연결될 수 있다. 모바일 에이전트가 진짜 개인화되려면 이 레이어가 반드시 필요하다.

마지막으로 실행 환경 레이어가 있다.  
이 레이어는 실제로 어디서, 어떤 권한으로, 어떤 런타임 위에서 에이전트가 작동하는지를 의미한다. 예를 들어 스마트폰의 Termux 환경, 원격 서버, Web UI, 메신저 채널, ADB 기반 단말 제어, 각종 API 연동 환경이 모두 여기에 포함된다.

정리하면 모바일 에이전트 구조는 다음과 같이 요약할 수 있다.

- 인터페이스: 사용자의 요청이 들어오는 입구
- 오케스트레이션: 작업 흐름을 설계하고 조율하는 층
- 모델: 이해, 추론, 생성 기능을 담당하는 두뇌
- 도구/스킬: 실제 작업을 수행하는 손발
- 메모리/지식: 개인화와 문맥 지속성을 위한 기억 체계
- 실행 환경: 실제 동작이 일어나는 런타임과 권한 기반

이 구조를 이해하면, 모바일 에이전트를 단순한 “챗봇 앱”이 아니라 하나의 플랫폼적 시스템으로 볼 수 있게 된다.

## 플랫폼 vs 프레임워크 vs 서비스

모바일 AI 에이전트를 설계할 때 자주 혼동되는 개념이 있다. 바로 플랫폼, 프레임워크, 서비스다.  
겉으로 보면 비슷해 보이지만, 각각이 지향하는 역할은 분명히 다르다. 이 차이를 이해하는 것은 어떤 수준에서 시스템을 만들고 있는지 판단하는 데 매우 중요하다.

먼저 서비스는 가장 익숙한 개념이다.  
사용자가 어떤 목적을 해결하기 위해 직접 이용하는 완성형 기능 묶음이라고 볼 수 있다. 예를 들어 “회의 요약 서비스”, “브리핑 생성 서비스”, “스마트폰 설정 자동화 서비스”는 모두 서비스의 예가 될 수 있다. 서비스는 사용자의 문제를 바로 해결하는 최종 산출물에 가깝다.

반면 프레임워크는 서비스를 만들기 위한 틀이다.  
개발자가 반복적으로 사용하는 구성 방식, 설계 원칙, 재사용 가능한 구성요소를 제공한다. 예를 들어 특정 에이전트를 만들 때 어떤 식으로 스킬을 정의하고, 어떤 방식으로 도구를 연결하고, 어떤 이벤트 구조를 사용할지를 정리한 패턴이 있다면 그것은 프레임워크라고 볼 수 있다.

플랫폼은 그보다 더 큰 개념이다.  
플랫폼은 여러 서비스와 프레임워크가 올라갈 수 있는 기반 환경이다. 사용자, 개발자, 도구, 모델, 데이터, 인터페이스가 모두 연결될 수 있는 운영 기반이라고 보면 된다. 플랫폼은 하나의 기능을 제공하는 데서 그치지 않고, 새로운 기능과 서비스가 계속 확장될 수 있는 구조를 만든다.

모바일 AI 에이전트 관점에서 보면 이 세 가지는 다음처럼 구분할 수 있다.

- 서비스: 실제 사용자가 쓰는 결과물
- 프레임워크: 그런 서비스를 만들기 위한 설계 틀
- 플랫폼: 여러 서비스와 프레임워크가 작동하는 실행 기반

예를 들어 사용자가 텔레그램으로 “내일 일정 브리핑해줘”라고 말해 결과를 받는 것은 서비스 경험이다.  
그 브리핑 기능이 일정 조회, 뉴스 수집, 요약, 메시지 발송의 패턴으로 반복 설계되어 있다면 그것은 프레임워크적 성격을 가진다. 그리고 그 모든 것이 OpenClaw, RAG, 툴, 메신저, 스마트폰 실행 환경 위에서 동작한다면, 그 전체는 플랫폼이라고 부를 수 있다.

왜 이 구분이 중요할까?  
많은 프로젝트가 서비스를 하나 만들고 나서 그것을 플랫폼이라고 부르지만, 실제로는 확장성과 구조가 없는 경우가 많기 때문이다. 진짜 플랫폼은 새로운 기능이 추가되어도 구조가 무너지지 않고, 다양한 인터페이스와 도구, 모델을 연결할 수 있어야 한다.

결국 우리가 구축하려는 모바일 AI 에이전트는 단순한 서비스 하나가 아니라, 여러 서비스와 스킬을 얹을 수 있는 플랫폼이어야 한다. 그리고 그 플랫폼을 체계적으로 확장하기 위해 프레임워크적 사고가 함께 필요하다.

## OpenClaw 기반 구조

이 책에서 다루는 모바일 에이전트 플랫폼의 핵심 기반은 OpenClaw다.  
OpenClaw는 단순히 모델을 호출하는 래퍼가 아니라, 에이전트를 실제 환경과 연결해 동작시키는 실행 플랫폼에 가깝다. 메신저, 파일 시스템, 웹 도구, cron, 세션, 스킬, 외부 런타임을 하나의 흐름으로 묶어주기 때문에 모바일 에이전트 구조를 구현하기에 적합하다.

OpenClaw 기반 구조를 이해할 때 가장 중요한 점은, 이것이 “대화창 안에 갇힌 AI”가 아니라는 사실이다.  
사용자는 텔레그램 같은 메신저에서 요청할 수도 있고, Streamlit 같은 Web UI에서 요청할 수도 있으며, 때로는 내부 스케줄러나 자동화 트리거에 의해 작업이 시작될 수도 있다. 즉 요청의 입구가 다양하고, OpenClaw는 그 입구들을 하나의 실행 레이어로 연결해준다.

그 위에서 에이전트는 모델을 사용해 의도를 파악하고, 필요한 도구를 선택하며, 스킬을 호출하고, 작업을 진행한다.  
예를 들어 문서 정리 요청이 들어오면 파일 읽기 도구와 요약 모델, 저장 도구가 연결될 수 있고, 스마트폰 제어 요청이 들어오면 ADB 또는 단말 제어 도구가 연결될 수 있다. OpenClaw는 이런 작업을 단순한 스크립트 호출이 아니라 에이전트 흐름으로 묶어낸다.

또 하나 중요한 특징은 확장성이다.  
OpenClaw는 Skills 구조를 통해 새로운 행동 패턴을 추가할 수 있고, 도구 구성을 바꿔 특정 환경에 맞는 실행 전략을 설계할 수 있다. 여기에 OpenDocuments 같은 지식 검색 구조를 연결하면, 에이전트는 단순 실행자에 그치지 않고 문맥을 이해하는 작업 수행자로 발전할 수 있다.

모바일 환경에서 특히 흥미로운 부분은 인터페이스와 실행 환경의 결합이다.  
예를 들어 Termux 위에서 OpenClaw를 실행하고, 텔레그램을 인터페이스로 사용하며, Web UI를 보조 관리 화면으로 두고, 필요한 경우 스마트폰을 직접 제어하는 툴을 붙이면 하나의 모바일 에이전트 플랫폼이 완성된다. 이 구조에서 핵심은 각각의 요소가 따로 존재하는 것이 아니라, 하나의 목표 지향형 흐름으로 연결된다는 점이다.

OpenClaw 기반 구조를 한 줄로 요약하면 이렇다.  
**모델, 도구, 스킬, 메모리, 인터페이스, 자동화를 하나의 실행 구조로 묶어주는 에이전트 플랫폼**. 바로 이 점 때문에 OpenClaw는 모바일 에이전트 실험을 넘어, 실제 운영 가능한 플랫폼의 토대로 볼 수 있다.

## 정리

모바일 에이전트 플랫폼을 이해하려면 에이전트를 단순한 AI 모델이나 챗봇으로 보지 말아야 한다.  
인터페이스, 오케스트레이션, 모델, 도구, 스킬, 메모리, 실행 환경이 함께 결합된 다층 구조로 봐야 한다. 또한 서비스, 프레임워크, 플랫폼의 차이를 구분해야만 우리가 무엇을 만들고 있는지 정확히 이해할 수 있다.

이 장의 핵심은 분명하다.  
우리가 구축하려는 것은 기능 하나가 아니라, 여러 기능을 유기적으로 얹을 수 있는 모바일 AI 에이전트 플랫폼이다. 그리고 OpenClaw는 그 구조를 실제로 구현하는 데 필요한 실행 중심 토대를 제공한다. 이후 장에서는 이 구조를 바탕으로 실행 환경, RAG, 도구, 스킬, 인터페이스를 하나씩 구체화해 나가게 된다.
