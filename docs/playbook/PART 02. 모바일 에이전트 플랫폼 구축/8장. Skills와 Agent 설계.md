## 스킬 기반 에이전트 구조

모바일 AI 에이전트를 구축할 때 가장 먼저 부딪히는 문제 중 하나는 복잡성이다.  
에이전트에게 할 일이 많아질수록, 단순히 하나의 거대한 프롬프트나 거대한 스크립트로 모든 것을 처리하려는 유혹이 생긴다. 하지만 이런 방식은 금방 유지보수 한계에 부딪힌다. 기능이 조금만 늘어나도 어디서 어떤 로직이 동작하는지 파악하기 어려워지고, 한 부분을 수정하면 다른 부분이 깨지기 쉽다. 그래서 에이전트 설계에서는 기능을 ‘스킬’ 단위로 나누는 접근이 매우 중요하다.

스킬은 쉽게 말해 에이전트가 수행할 수 있는 재사용 가능한 작업 단위다.  
예를 들어 뉴스 브리핑 생성, 회의록 요약, 스마트폰 설정 진입, 특정 앱 실행, 문서 검색, 일정 확인, 메시지 전송 같은 작업들은 각각 하나의 스킬로 분리할 수 있다. 중요한 점은 스킬이 단순한 명령 목록이 아니라, 특정 목적을 달성하기 위한 하나의 잘 정의된 능력이라는 점이다.

스킬 기반 구조가 중요한 이유는 에이전트를 ‘기능의 집합’이 아니라 ‘능력의 조합’으로 만들 수 있기 때문이다.  
에이전트는 사용자의 요청을 받았을 때, 모든 것을 처음부터 추론하는 대신, 이미 정의된 스킬 중 무엇을 조합해야 하는지를 판단한다. 이 구조는 추론 비용을 줄여주고, 동작의 예측 가능성을 높이며, 테스트와 유지보수를 훨씬 쉽게 만든다.

예를 들어 “내일 아침 브리핑 준비해줘”라는 요청은 겉보기에는 하나의 문장이지만, 실제로는 여러 스킬의 조합으로 이뤄질 수 있다.

- 일정 조회 스킬
- 뉴스 수집 스킬
- 요약 스킬
- 메시지 발송 스킬

이처럼 고수준 요청을 스킬 조합으로 처리하면, 하나의 요청이 여러 능력의 연결로 해석된다. 즉 에이전트는 단순히 대답하는 존재가 아니라, 상황에 맞는 능력을 선택하고 실행하는 조정자로 진화한다.

스킬 기반 구조는 또 다른 이점을 준다.  
바로 이식성과 확장성이다. 특정 스킬은 메신저 인터페이스에서도 쓸 수 있고, Web UI에서도 쓸 수 있으며, cron 자동화에도 그대로 재사용할 수 있다. 인터페이스가 바뀌어도 내부 스킬이 유지되면, 에이전트는 더 쉽게 확장된다. 이것은 모바일 AI 에이전트를 일회성 데모가 아니라 플랫폼으로 성장시키는 데 매우 중요한 조건이다.

결국 스킬 기반 설계는 에이전트를 더 똑똑하게 만드는 기술이 아니라, 더 구조적으로 일하게 만드는 설계 방식이다.  
에이전트가 장기적으로 유용하려면, 많은 기능을 무작정 붙이는 것보다, 능력을 스킬 단위로 정리하고 조합 가능한 구조를 먼저 만드는 것이 훨씬 중요하다.

## 작업 분해 방법

에이전트 설계에서 가장 어려우면서도 중요한 능력은 작업을 잘게 나누는 것이다.  
사용자는 보통 목표 단위로 말한다. “회의 준비해줘”, “문서 정리해줘”, “아침 루틴 자동화해줘”, “앱 설치해줘”처럼 말이다. 하지만 실행 시스템은 이런 목표를 그대로 처리할 수 없다. 실제로는 목표를 더 작은 단계로 분해하고, 각 단계를 어떤 스킬이나 도구로 처리할지 결정해야 한다.

좋은 작업 분해는 세 가지 특징을 가진다.  
첫째, 단계가 충분히 작아야 한다. 너무 큰 단위는 실패했을 때 원인을 알기 어렵다.  
둘째, 각 단계가 명확한 출력 조건을 가져야 한다. 어떤 상태가 되면 그 단계가 끝났다고 볼 수 있는지가 분명해야 한다.  
셋째, 단계 간 의존 관계가 드러나야 한다. 무엇이 먼저 되어야 하고, 무엇이 그 다음에 올 수 있는지가 보여야 한다.

예를 들어 “회의 준비해줘”라는 요청을 생각해보자. 이 목표를 그냥 하나의 작업으로 처리하려 하면 에이전트는 막연해진다. 하지만 이를 다음처럼 분해하면 훨씬 다루기 쉬워진다.

1. 내일 일정에서 회의 목록 확인
2. 회의 관련 메일 및 자료 검색
3. 핵심 정보 요약
4. 필요한 브리핑 문안 생성
5. 결과를 메신저 또는 문서로 전달

이렇게 나누면 각 단계는 서로 연결되지만, 독립적으로 테스트하고 검증할 수 있다. 그리고 특정 단계만 개선하거나 교체하는 것도 쉬워진다.

작업 분해는 단순한 편의가 아니라, 에이전트의 신뢰성과 직결된다.  
잘게 쪼개진 작업은 실패를 더 빨리 감지할 수 있고, 어디에서 문제가 생겼는지 추적하기 쉽다. 또한 사용자가 어느 지점에서 개입해야 하는지도 명확해진다. 예를 들어 브리핑 문안 생성까지는 자동으로 하고, 최종 발송 전에는 사용자 확인을 받는 식의 설계가 가능해진다.

여기서 중요한 것은 작업 분해가 항상 기술적 순서와 같지는 않다는 점이다.  
사용자의 목표, 리스크, 자동화 가능성, 도구 제약, 승인 필요 여부를 함께 고려해야 한다. 즉 작업 분해는 단순한 프로그래밍 문제라기보다, 인간의 의도를 시스템 실행 구조로 번역하는 설계 작업에 가깝다.

모바일 에이전트에서는 이 능력이 특히 더 중요하다.  
스마트폰 환경은 화면 전환, 권한 팝업, 네트워크 상태, 앱 버전 차이처럼 변수가 많기 때문이다. 따라서 하나의 거대한 작업보다, 짧고 검증 가능한 단계의 연속으로 설계하는 것이 훨씬 안정적이다. 에이전트가 진짜 일 잘하는 것처럼 보이려면, 실제로는 이런 세밀한 분해 구조가 뒤에 있어야 한다.

## 하네스(Harness) 설계

스킬과 작업 분해가 준비되었다면, 이제 그것들을 어떻게 감싸고 운영할 것인지가 중요해진다.  
여기서 등장하는 개념이 하네스(Harness)다. 일반적으로 하네스는 여러 요소를 안정적으로 묶고 제어하기 위한 보조 구조를 뜻한다. 에이전트 설계에서 하네스는 스킬, 모델, 도구, 입력, 출력, 예외 처리, 상태 관리를 하나의 실행 틀 안에 담는 운영 구조라고 이해할 수 있다.

쉽게 말하면 스킬이 개별 능력이라면, 하네스는 그 능력들이 실제 상황에서 흔들리지 않고 동작하도록 붙잡아주는 프레임이다.  
예를 들어 같은 “문서 요약” 스킬이라도 메신저에서 호출될 때와 Web UI에서 호출될 때, 혹은 cron 자동화로 실행될 때 입력 방식과 출력 방식이 달라질 수 있다. 이때 하네스가 있으면 공통 규칙을 유지하면서도 인터페이스별 차이를 유연하게 흡수할 수 있다.

좋은 하네스는 보통 다음 요소를 포함한다.

- 입력 규격 정의: 어떤 입력을 받아야 하는가
- 실행 조건 정의: 어떤 상황에서 실행 가능한가
- 상태 추적: 진행 중, 완료, 실패 상태를 어떻게 기록할 것인가
- 예외 처리: 실패 시 어디서 멈추고 어떻게 복구할 것인가
- 출력 규격 정의: 결과를 어떤 형식으로 반환할 것인가
- 로그/관측성: 무엇이 일어났는지 어떻게 추적할 것인가

이 구조가 중요한 이유는 에이전트를 “잘 돌아가는 데모”에서 “운영 가능한 시스템”으로 바꿔주기 때문이다.  
스킬만 있으면 기능은 만들 수 있지만, 운영은 어렵다. 반면 하네스가 있으면 같은 스킬을 여러 환경에서 안정적으로 돌릴 수 있고, 실패를 기록하고, 상태를 추적하고, 인터페이스를 바꿔도 일관된 실행 경험을 유지할 수 있다.

모바일 AI 에이전트에서는 특히 하네스의 가치가 크다.  
왜냐하면 모바일 환경은 변수가 많기 때문이다. 네트워크가 끊길 수도 있고, 앱이 백그라운드에서 종료될 수도 있고, 단말 상태가 달라질 수도 있으며, 특정 작업은 사용자 승인이 필요할 수도 있다. 이런 환경에서는 스킬 자체보다, 그 스킬을 어떻게 보호하고 관리하며 반복 가능한 흐름으로 만드는지가 더 중요하다.

예를 들어 스마트폰 설정 자동화 하네스를 생각해볼 수 있다.  
입력은 “변경할 설정 값”, 실행 조건은 “ADB 연결 상태 정상”, 중간 상태는 “설정 화면 진입 완료 / 대상 메뉴 진입 완료 / 값 변경 완료”, 실패 처리 방식은 “3회 재시도 후 사용자 확인 요청”처럼 정의할 수 있다. 이렇게 되면 스킬 하나가 아니라, 운영 가능한 실행 흐름이 된다.

결국 하네스 설계는 에이전트에 ‘규율’을 부여하는 작업이다.  
에이전트가 단순히 똑똑한 답을 내는 것이 아니라, 예측 가능하고 반복 가능하며 복구 가능한 방식으로 행동하게 만드는 구조다. 그리고 이 구조가 있어야만 모바일 AI 에이전트는 실험실 밖의 현실 환경에서도 지속적으로 동작할 수 있다.

## 정리

Skills와 Agent 설계의 핵심은 복잡한 목표를 잘 정의된 능력과 실행 흐름으로 바꾸는 것이다.  
스킬 기반 구조는 에이전트를 조합 가능한 능력의 집합으로 만들고, 작업 분해는 사용자의 목표를 실행 가능한 단계로 변환하며, 하네스는 그 흐름을 운영 가능한 시스템으로 안정화한다.

결국 잘 설계된 에이전트는 “모든 것을 한 번에 해결하는 거대한 AI”가 아니다.  
오히려 작은 능력들을 구조적으로 엮고, 각 단계를 명확히 정의하며, 예외와 복구까지 포함하는 정교한 실행 시스템이다. 그리고 바로 이 구조가 모바일 AI 에이전트를 단순한 자동화 스크립트가 아니라, 확장 가능한 플랫폼으로 성장시키는 핵심 기반이 된다.
