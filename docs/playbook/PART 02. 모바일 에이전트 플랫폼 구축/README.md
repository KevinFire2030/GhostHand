# PART 02. 모바일 에이전트 플랫폼 구축

이 파트는 모바일 AI 에이전트를 실제로 구축하기 위해 필요한 핵심 구성요소를 다룬다.  
PART 1이 왜 모바일 AI 에이전트가 필요한지를 설명하는 개념적 파트였다면, PART 2는 그것을 실제 시스템으로 만들기 위한 구조와 실행 기반을 설명하는 실전 파트에 해당한다.

핵심 질문은 단순하다.  
**모바일 에이전트는 어떤 구조로 설계되고, 어떤 실행 환경 위에서, 어떤 두뇌와 도구와 인터페이스를 통해 실제로 동작하는가?**

---

## 4장 요약. 전체 아키텍처 개요

4장은 모바일 에이전트를 단일 모델이나 챗봇이 아니라, 여러 계층이 결합된 시스템으로 바라보게 만든다.  
이 장에서는 인터페이스, 오케스트레이션, 모델, 도구/스킬, 메모리, 실행 환경으로 이어지는 다층 구조를 설명한다.

핵심은 다음과 같다.
- 모바일 에이전트는 하나의 모델이 아니라 여러 레이어가 결합된 실행 시스템이다.
- 서비스, 프레임워크, 플랫폼을 구분해야 한다.
- OpenClaw는 단순 챗봇이 아니라, 도구·스킬·세션·자동화를 묶는 실행 플랫폼이다.

즉 이 장은 이후 모든 장의 기반이 되는 구조적 시각을 제공한다.

---

## 5장 요약. 실행 환경 구축 (Termux + OpenClaw)

5장은 스마트폰 위에서 에이전트를 실제로 돌리기 위한 실행 환경을 다룬다.  
Termux를 통해 모바일 기기 안에 리눅스에 가까운 실행 기반을 만들고, 그 위에서 OpenClaw 같은 플랫폼을 어떻게 운영할 수 있는지 설명한다.

핵심 내용은 다음과 같다.
- Termux는 모바일 안의 개발/실행 서버 같은 역할을 한다.
- OpenClaw는 범용 플랫폼 지향, PicoClaw는 목적 특화형 경량 구조로 이해할 수 있다.
- 모바일 AI 실행은 원격 호출형, 하이브리드형, 온디바이스형을 혼합하는 방향으로 발전할 가능성이 크다.

특히 이 장은 “모바일에서 AI를 실행한다”는 말이 단순 앱 사용이 아니라, 스마트폰을 실행 허브로 만드는 구조라는 점을 강조한다.

---

## 6장 요약. 에이전트의 두뇌 만들기 (RAG)

6장은 모바일 에이전트가 단순히 똑똑한 모델이 아니라, **근거를 가진 실무형 두뇌**를 가지려면 무엇이 필요한지를 설명한다.  
여기서 핵심은 RAG, 즉 Retrieval-Augmented Generation 구조다.

핵심 내용은 다음과 같다.
- RAG는 외부 문서를 검색해 모델 생성과 결합하는 구조다.
- 이 구조를 통해 최신성, 정확성, 개인화, 출처 기반 설명이 가능해진다.
- OpenDocuments는 GitHub, Notion, Google Drive 등 여러 문서 소스를 연결하고 cited answer를 제공하는 실전형 RAG 도구다.
- 개인 지식베이스는 에이전트 품질을 결정하는 핵심 자산이다.

즉 이 장은 에이전트의 ‘기억’과 ‘근거’를 어떻게 설계할 것인지에 대한 실질적인 방향을 제시한다.

---

## 7장 요약. Tools: 스마트폰을 제어하는 힘

7장은 모바일 에이전트를 “말하는 시스템”에서 “행동하는 시스템”으로 바꾸는 핵심인 도구와 단말 제어를 다룬다.  
Wireless ADB 구조를 중심으로, 스마트폰 제어가 어떻게 가능해지고 어떤 전략이 필요한지를 설명한다.

핵심 내용은 다음과 같다.
- Wireless ADB는 스마트폰 제어의 핵심 실행 채널이다.
- 단말 제어는 시스템 레벨, 앱 레벨, UI 레벨 전략으로 나눠 볼 수 있다.
- API 기반 제어가 가능하면 우선 사용하고, 직접 UI 제어는 필요한 경우에만 보조적으로 쓰는 것이 바람직하다.

즉 이 장은 모바일 에이전트의 손발이 어떻게 만들어지는지 보여준다.

---

## 8장 요약. Skills와 Agent 설계

8장은 에이전트를 거대한 하나의 AI로 보는 대신, **작은 능력의 조합체**로 설계하는 관점을 설명한다.  
여기서 핵심 개념은 스킬, 작업 분해, 하네스 설계다.

핵심 내용은 다음과 같다.
- 스킬은 재사용 가능한 작업 단위다.
- 사용자의 목표는 더 작은 단계로 잘게 분해되어야 한다.
- 하네스는 스킬, 상태, 입력, 출력, 예외 처리를 하나의 운영 구조로 묶는 틀이다.

즉 이 장은 에이전트를 구조적으로 일하게 만드는 설계법을 다룬다.

---

## 9장 요약. 인터페이스 만들기

9장은 사용자가 모바일 AI 에이전트와 어떻게 만나는지를 설명한다.  
Web UI, 메신저, 음성 인터페이스가 각각 어떤 역할을 하는지, 그리고 왜 하나의 채널이 아니라 다중 인터페이스가 중요한지를 보여준다.

핵심 내용은 다음과 같다.
- Streamlit 기반 Web UI는 설정, 상태 확인, 결과 검토, 다운로드 같은 운영 콘솔 역할에 적합하다.
- 텔레그램/슬랙 같은 메신저는 가장 자연스러운 명령 인터페이스다.
- 음성 인터페이스는 모바일 환경에서 가장 자연스러운 입력 확장 방식이다.
- 좋은 플랫폼은 하나의 인터페이스만 고집하지 않고, 서로 다른 채널을 상황에 맞게 결합한다.

즉 이 장은 에이전트의 입구와 출구, 그리고 사용자와의 실제 접점을 설계하는 문제를 다룬다.

---

## PART 2의 핵심 정리

PART 2 전체를 관통하는 핵심 메시지는 명확하다.

1. 모바일 에이전트는 단순한 챗봇이 아니라 다층 구조의 실행 시스템이다.  
2. 이 시스템은 실행 환경, RAG 기반 두뇌, 스마트폰 제어 도구, 스킬 구조, 인터페이스까지 모두 함께 설계되어야 한다.  
3. OpenClaw는 그 중심에서 실행 플랫폼 역할을 하고, OpenDocuments 같은 도구는 지식과 기억 계층을 담당한다.

결국 PART 2는 “모바일 AI 에이전트를 실제로 어떻게 만들 것인가”에 대한 구조적 답변이다.  
이 파트를 통해 독자는 모바일 에이전트가 단순한 아이디어가 아니라, 이미 구현 가능한 플랫폼적 조합이라는 사실을 이해하게 된다.
