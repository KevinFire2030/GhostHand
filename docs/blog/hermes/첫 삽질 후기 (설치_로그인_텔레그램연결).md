# Hermes Agent 첫 삽질 후기: WSL 설치부터 Codex 로그인, Telegram 연결까지

> 작성일: 2026-05-06  
> 환경: Windows 11 + WSL2 Ubuntu 26.04 LTS  
> 목표: Windows 환경에서 Hermes Agent를 설치하고, OpenAI Codex 모델과 Telegram 봇까지 연결하기

## 0. 결론부터

오늘의 결론은 단순하다.

**Windows에서 Hermes Agent를 쓰려면 WSL2 Ubuntu 위에 설치하는 흐름이 가장 현실적이다.**

그리고 설치 자체보다 더 시간을 잡아먹은 부분은 다음 세 가지였다.

- WSL 서비스가 멈춰서 `wsl --status`, `wsl -l -v`가 반응하지 않음
- Codex 로그인에서 `device-auth`가 계정 보안 설정 때문에 막힘
- WSL 안에서 `codex` 명령이 Windows 쪽 npm 경로를 타면서 꼬임

그래도 최종적으로는 성공했다.

```text
Windows
└─ WSL2 Ubuntu
   └─ Hermes Agent
      ├─ OpenAI Codex / gpt-5.5
      └─ Telegram Bot DM 연결
```

## 1. 시작: Hermes 공식 GitHub 확인

먼저 Hermes Agent의 공식 저장소를 확인했다.

- 공식 GitHub: https://github.com/NousResearch/hermes-agent
- 공식 문서: https://hermes-agent.nousresearch.com/docs/

README 기준으로 Hermes Agent는 Linux, macOS, WSL2, Android Termux를 지원한다. 반대로 Windows 네이티브 설치는 공식 지원하지 않는다.

따라서 Windows 사용자는 WSL2 위에서 설치하는 것이 기본 경로다.

## 2. WSL부터 삽질 시작

처음에는 PowerShell에서 WSL 상태를 확인했다.

```powershell
wsl --status
```

그런데 아무 반응이 없었다.

처음에는 WSL이 안 깔린 줄 알았지만, 다음 명령은 정상으로 나왔다.

```powershell
wsl --version
```

출력은 대략 이랬다.

```text
WSL 버전: 2.5.10.0
커널 버전: 6.6.87.2-1
WSLg 버전: 1.0.66
Windows 버전: 10.0.26200.7462
```

즉 WSL 자체는 설치되어 있었다.

문제는 `wsl -l -v`도 멈춘다는 점이었다.

```powershell
wsl -l -v
```

처음에는 예전 방식대로 `LxssManager` 서비스를 재시작하려 했다.

```powershell
Restart-Service LxssManager
```

하지만 이런 오류가 나왔다.

```text
서비스 이름이 'LxssManager'인 서비스를 찾을 수 없습니다.
```

최근 Store/MSIX 방식 WSL에서는 `LxssManager` 대신 `WslService`가 보일 수 있다.

확인해보니 실제로 `WslService`가 있었다.

```powershell
Get-Service *wsl*
```

```text
Status   Name        DisplayName
------   ----        -----------
Running  WSLService  WSL Service
```

그런데 문제는 이 서비스를 재시작할 때였다.

```powershell
Restart-Service WslService
```

서비스가 중지 단계에서 계속 기다리는 상태가 되었다.

```text
경고: 'WSL Service(WslService)' 서비스가 중지될 때까지 기다리는 중...
```

이 상태에서는 더 붙잡기보다 재부팅이 답이었다.

재부팅 후 다시 확인했다.

```powershell
Get-Service WslService
wsl -l -v
wsl --status
```

결과는 정상으로 돌아왔다.

```text
NAME              STATE     VERSION
* docker-desktop  Running   2
  Ubuntu          Stopped   2
```

다만 기본 배포판이 `docker-desktop`으로 잡혀 있었다.

```text
기본 배포: docker-desktop
기본 버전: 2
```

그래서 Ubuntu를 기본 배포판으로 바꿨다.

```powershell
wsl --set-default Ubuntu
```

## 3. Ubuntu 첫 실행과 root 접속 문제

`wsl`을 실행하니 Ubuntu 첫 설정이 시작됐다.

```powershell
wsl
```

사용자 계정을 만들었다.

```text
Create a default Unix user account: fireman_pro
New password:
Retype new password:
passwd: password updated successfully
```

그런데 첫 실행 후 프롬프트가 이상했다.

```bash
#
```

일반 사용자라면 보통 이런 식이어야 한다.

```bash
fireman_pro@GenAI-Nexus:~$
```

중간에 이런 에러도 한 번 보였다.

```text
WSL ERROR: getpwuid(1000) failed
```

다행히 사용자 계정 자체는 생성되어 있었다.

```powershell
wsl -d Ubuntu -u fireman_pro
```

Ubuntu 안에서 확인했다.

```bash
whoami
```

```text
fireman_pro
```

이후 기본 사용자를 고정하기 위해 `/etc/wsl.conf`를 설정했다.

```bash
sudo tee /etc/wsl.conf <<'EOF'
[user]
default=fireman_pro
EOF
```

그리고 PowerShell로 돌아와 WSL을 종료했다.

```powershell
wsl --shutdown
```

다시 `wsl`로 들어갔을 때 `whoami`가 `fireman_pro`로 나오면 정상이다.

## 4. Ubuntu 업데이트

Ubuntu가 정상으로 들어가진 뒤 기본 업데이트를 진행했다.

```bash
sudo apt update && sudo apt upgrade -y
```

업데이트는 정상 완료됐다.

```text
12 packages can be upgraded.
Processing triggers for libc-bin ...
Processing triggers for man-db ...
```

선택사항으로 매일 뜨는 Ubuntu 안내문을 숨기려면 다음 파일을 만들면 된다.

```bash
touch ~/.hushlogin
```

## 5. Hermes Agent 설치

이제 Hermes 설치를 진행했다.

먼저 홈 디렉터리로 이동했다.

```bash
cd ~
```

기본 도구를 설치했다.

```bash
sudo apt update
sudo apt install -y curl git build-essential python3 python3-venv
```

공식 설치 명령을 실행했다.

```bash
curl -fsSL https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.sh | bash
```

설치 후 셸 설정을 다시 불러왔다.

```bash
source ~/.bashrc
```

그리고 초기 설정을 시작했다.

```bash
hermes setup
```

설정 방식은 `Quick setup`을 선택했다.

```text
Quick setup — provider, model & messaging (recommended)
```

처음에는 핵심 설정만 잡고, 세부 설정은 나중에 바꾸는 편이 안전하다.

## 6. Codex 로그인 삽질

Hermes 설정 중 inference provider 단계에서 OpenAI Codex 로그인이 시작됐다.

처음에는 device code 방식으로 로그인하라고 나왔다.

```text
Open this URL in your browser:
https://auth.openai.com/codex/device

Enter this code:
....
```

하지만 브라우저에서 인증하려고 하니 다음 안내가 나왔다.

```text
ChatGPT 보안 설정 내 Codex용 장치 코드 인증을 활성화한 뒤,
"codex login --device-auth"를 다시 실행하세요.
```

문제는 ChatGPT 보안 설정 화면에 `Codex용 장치 코드 인증 활성화` 문구는 보이는데, 실제 활성화 버튼이나 토글이 보이지 않았다는 점이다.

회사/조직 계정에서는 정책 때문에 해당 토글이 숨겨지거나 막힐 수 있다. 개인 계정에서도 UI 롤아웃 상태에 따라 보이지 않을 수 있다.

여기서 시간을 많이 썼다.

결국 우회 방법은 간단했다.

**`--device-auth`가 아니라 일반 로그인 플로우를 쓰면 된다.**

```bash
codex login
```

이 명령은 WSL 안에서 로컬 로그인 서버를 열고 브라우저 OAuth 흐름을 사용한다.

```text
Starting local login server on http://localhost:1455.
If your browser did not open, navigate to this URL to authenticate:
https://auth.openai.com/oauth/authorize?...
```

브라우저에서 로그인 후 터미널에 다음 메시지가 나왔다.

```text
Successfully logged in
```

이 순간 가장 큰 산을 넘었다.

## 7. WSL 안에서 Codex CLI 경로가 꼬인 문제

중간에 Codex CLI도 한 번 꼬였다.

WSL에서 다음 명령을 실행했다.

```bash
codex login --device-auth
```

처음에는 이런 오류가 났다.

```text
/mnt/c/Users/fireman_pro/AppData/Roaming/npm/codex: 15: exec: node: not found
```

이건 WSL 안의 Codex가 아니라 Windows 쪽 npm 전역 실행 파일을 잡고 있었다는 뜻이다.

확인해보니 `npm`의 global prefix가 Windows 경로로 되어 있었다.

```bash
npm config get prefix
```

처음에는 이런 식이었다.

```text
C:\Users\fireman_pro\AppData\Roaming\npm
```

이러면 WSL에서 `npm install -g`를 해도 Windows 쪽에 설치된다. 그래서 Linux용 optional dependency가 없다는 에러도 났다.

```text
Error: Missing optional dependency @openai/codex-linux-x64.
```

이후 Hermes가 잡아둔 WSL 내부 Node prefix를 사용하게 되면서 정상화됐다.

```bash
npm config get prefix
```

```text
/home/fireman_pro/.hermes/node
```

Codex CLI를 다시 설치했다.

```bash
npm install -g @openai/codex@latest
hash -r
which codex
codex --version
```

결과는 정상.

```text
/usr/local/bin/codex
codex-cli 0.128.0
```

그리고 일반 로그인으로 성공했다.

```bash
codex login
```

```text
Successfully logged in
```

## 8. Hermes가 Codex 자격증명을 가져오기

Codex CLI 로그인이 성공한 뒤 Hermes 설정으로 돌아갔다.

Hermes가 기존 Codex CLI 자격증명을 발견했다.

```text
Found existing Codex CLI credentials at ~/.codex/auth.json
Hermes will create its own session to avoid conflicts with Codex CLI / VS Code.
Import these credentials? (a separate login is recommended) [y/N]:
```

여기서는 `y`를 선택했다.

```text
Credentials imported.
Config updated: /home/fireman_pro/.hermes/config.yaml (model.provider=openai-codex)
```

이후 기본 모델 선택 화면이 나왔다.

```text
Select default model:
-> gpt-5.5
   gpt-5.4
   gpt-5.4-mini
   gpt-5.3-codex
   gpt-5.2
```

기본 모델은 `gpt-5.5`를 선택했다.

최종적으로 Hermes의 provider는 `openai-codex`, 기본 모델은 `gpt-5.5`로 잡혔다.

## 9. Telegram 봇 만들기

이제 Telegram 연결 단계다.

BotFather에서 새 봇을 만들었다.

```text
/newbot
```

봇 이름과 username을 정한 뒤 BotFather가 HTTP API token을 발급해준다.

중요한 점:

> Telegram bot token은 비밀번호와 같다.  
> 공개 채팅이나 GitHub에 절대 올리면 안 된다.

만약 실수로 노출했다면 BotFather에서 즉시 재발급해야 한다.

```text
/mybots → 봇 선택 → API Token → Revoke current token
```

Hermes 설정에서 Telegram bot token을 입력했다.

처음에는 잘못된 형식의 토큰을 넣어서 오류가 났다.

```text
Invalid token format. Expected: <numeric_id>:<alphanumeric_hash>
```

정상 토큰을 넣자 저장됐다.

```text
Telegram token saved
```

## 10. Telegram Allowed User ID 설정

Hermes는 봇을 아무나 쓰지 못하게 allowed user ID를 설정할 수 있다.

설정 화면에는 이런 안내가 나온다.

```text
Allowed user IDs (comma-separated, leave empty for open access):
```

여기서 비워두면 봇이 사실상 열릴 수 있으므로, 반드시 내 Telegram user ID를 넣는 것이 좋다.

Telegram user ID는 `@userinfobot`에게 메시지를 보내면 확인할 수 있다.

예시:

```text
5440299023
```

이 값을 allowed user IDs에 넣었다.

## 11. DM chat_id와 그룹 chat_id

Telegram에서 개인 DM의 `chat_id`는 보통 user ID와 같다.

예시:

```text
DM chat_id: 5440299023
```

그룹 chat_id는 음수로 시작하는 경우가 많다.

이번 테스트 그룹은 다음 값이었다.

```text
group chat_id: -1003844548929
```

포럼 토픽까지 구분해야 하는 경우에는 thread/topic ID도 필요하다.

```text
topic/thread_id: 33
```

정리하면 다음과 같다.

```text
Telegram user ID / DM chat_id: 5440299023
Telegram group chat_id: -1003844548929
Telegram topic/thread_id: 33
```

## 12. Telegram 연결 테스트

마지막으로 Hermes Telegram 봇에게 DM을 보냈다.

봇이 테스트 메시지를 보냈다.

```text
Hermes Telegram 연결 테스트 메시지입니다. ✅
```

사용자가 `된다`라고 답하자 Hermes 봇이 응답했다.

```text
좋아요, 다행이에요!
필요하면 다음 단계도 바로 도와드릴게요.
```

그리고 Hermes에게 어떤 에이전트인지 물었다.

```text
넌 어떤 AI 에이전트니?
```

Hermes는 자신을 Telegram에서 동작하는 Hermes 기반 작업형 AI 비서라고 설명했다.

이것으로 설치, 로그인, Telegram DM 연결까지 성공했다.

## 13. 오늘의 삽질 포인트 정리

### 1) `wsl --status`가 멈춘다고 WSL 미설치는 아니다

`wsl --version`이 나오면 WSL 자체는 설치되어 있다.

`wsl -l -v`나 `wsl --status`가 멈추면 WSL 서비스가 꼬였을 가능성이 크다.

확인 명령:

```powershell
Get-Service *wsl*
```

최근 환경에서는 `LxssManager`가 아니라 `WslService`가 보일 수 있다.

### 2) WSLService가 stopping에서 걸리면 재부팅이 빠르다

```powershell
Restart-Service WslService
```

이 명령이 중지 대기 상태에서 계속 멈추면, 괜히 오래 붙잡기보다 재부팅이 낫다.

### 3) 기본 배포판이 docker-desktop이면 Ubuntu로 바꾸자

```powershell
wsl --set-default Ubuntu
```

그렇지 않으면 `wsl` 입력 시 의도치 않게 Docker Desktop 배포판이 기본으로 잡힐 수 있다.

### 4) Ubuntu가 root로 열리면 기본 유저를 고정하자

```bash
sudo tee /etc/wsl.conf <<'EOF'
[user]
default=fireman_pro
EOF
```

이후 PowerShell에서:

```powershell
wsl --shutdown
wsl
whoami
```

### 5) `codex login --device-auth`가 막히면 `codex login`을 써보자

device auth는 ChatGPT 보안 설정에서 별도 활성화가 필요할 수 있다.

하지만 일반 브라우저 OAuth 플로우는 다음 명령으로 가능했다.

```bash
codex login
```

이번 설치에서는 이 방법이 정답이었다.

### 6) WSL에서 npm global prefix가 Windows 경로인지 확인하자

```bash
npm config get prefix
```

Windows 경로가 나오면 WSL 안에서 설치한 npm 패키지가 Windows 쪽에 깔릴 수 있다.

정상 예시:

```text
/home/fireman_pro/.hermes/node
```

### 7) Telegram bot token은 절대 공개하지 말자

Telegram bot token은 봇을 제어할 수 있는 비밀번호다.

노출했다면 BotFather에서 즉시 revoke 후 새 토큰을 발급해야 한다.

## 15. 마무리

이번 설치는 단순히 `curl | bash` 한 줄로 끝나는 작업은 아니었다.

WSL 서비스, Ubuntu 기본 사용자, npm 경로, Codex 인증 방식, Telegram token 보안까지 하나씩 걸렸다.

하지만 이 과정을 지나고 나면 구조는 꽤 깔끔하다.

```text
Windows는 호스트
WSL2 Ubuntu는 실행 환경
Hermes는 에이전트 런타임
Codex는 추론 모델 연결
Telegram은 모바일 인터페이스
```

즉, PC 안에만 갇힌 CLI 도구가 아니라 Telegram DM으로 불러서 작업시키는 개인 AI 에이전트 환경이 된다.

오늘의 한 줄 평:

> 설치는 조금 삽질이지만, 한 번 연결되면 꽤 든든한 개인 AI 작업 비서가 된다.
