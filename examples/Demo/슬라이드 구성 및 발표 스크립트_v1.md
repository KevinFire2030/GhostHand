# [슬라이드] #1 (1분)
## 제목
**1. 활용 기술**

## 핵심 문구
**세계 최초 온디바이스 AI 단말 제어 기술**

## 왼쪽
### 오픈클루
- 휴대폰 안에서 동작하는 AI 도우미
- 사람이 말하듯 명령하면 이해하고 실행
- 앱 실행, 화면 이동, 정보 확인 등을 대신 수행
- 별도 PC 없이 휴대폰 안에서 바로 동작

### 무선 단말 제어 기술
- 선 없이 휴대폰을 원격 제어하는 방식
- 버튼 누르기, 앱 열기, 화면 이동 같은 동작 가능
- 테스트용 단말을 사람이 직접 만지지 않아도 됨
- 휴대폰과 연결만 되면 같은 방식으로 제어 가능

## 오른쪽
### 시스템 구성도
- 사용자 명령
- 휴대폰 안의 AI
- 단말 제어
- 결과 확인

## 발표 스크립트
여기서 핵심 기술은 두 가지입니다.  
첫째는 **오픈클루**입니다. 쉽게 말하면, 휴대폰 안에서 돌아가는 AI 도우미입니다. 사람이 "앱 열어줘", "검증해줘"라고 말하면, 그 뜻을 이해하고 실제 동작으로 옮깁니다.  
둘째는 **무선 단말 제어 기술**입니다. 쉽게 말하면, 선 없이 휴대폰을 제어하는 방식입니다. 사람이 손으로 누르던 작업을, 연결된 상태에서 자동으로 수행할 수 있습니다.  
즉, 이 두 기술을 합치면 **휴대폰 안의 AI가 그 휴대폰을 직접 움직이는 구조**가 됩니다. 이 부분이 이번 PoC의 가장 중요한 포인트입니다.

# [슬라이드] #2 (2분)
## 제목
**2. 활용 예시: 테마 앱 검증 자동화**

## 상단
### As-is 워크플로우
**현재 구축 중인 방식**
- 테스트 단말에서 앱 설치
- 자동화 서버가 단말과 연결
- 테스트 스크립트 실행
- 결과 이미지를 대량 저장
- 클라우드에서 이미지를 다시 내려받아 판독
- 판정 결과를 별도 시스템에 업로드

## 하단
### To-be 워크플로우
**오픈클루 + 무선 단말 제어 방식**
- 사용자가 자연어로 검증 요청
- 단말 안의 AI가 검증 시나리오 작성
- 앱 다운로드 / 설치 / 실행
- 결과 저장
- 결과 판정
- 결과 시스템 업로드

## 강조 포인트
- 중간 서버 단계 축소
- 이미지 이동/재처리 단계 축소
- 사람 개입 단계 축소
- 테스트 단말 중심으로 절차 단순화

## 발표 스크립트
이 슬라이드는 현재 방식과 앞으로 가능한 방식을 비교한 것입니다.  
현재 방식은 테스트 단말만 있는 것이 아니라, 중간에 자동화 서버가 있고, 결과 이미지를 따로 저장하고, 클라우드에서 다시 판독하고, 마지막에 결과를 업로드하는 식으로 단계가 많이 나뉘어 있습니다.  
즉, **잘게 나뉜 여러 단계를 연결해서 돌리는 구조**입니다.  
반면 To-be는 훨씬 단순합니다. 사용자가 "이 테마 앱 검증해줘"라고 말하면, 휴대폰 안의 AI가 직접 시나리오를 만들고, 설치하고, 실행하고, 결과를 보고, 판정하고, 마지막 결과까지 넘길 수 있습니다.  
핵심은 **중간 단계가 줄어든다**, 그리고 **테스트의 중심이 서버가 아니라 단말 자체로 이동한다**는 점입니다.  
즉, 지금은 여러 시스템이 이어져야 하는 일을, 앞으로는 **단말 하나에서 훨씬 짧은 흐름으로 처리할 수 있다**는 것입니다.

# [슬라이드] #3 (1분)
## 제목
**3. 기대효과**

## 구성
### 테스트 간소화
- 테스트 단말에서 대부분의 절차 직접 수행
- 절차 단축
- 사람 손이 덜 감

### 비용 절감
- 개발비/운영비 축소
- 예시: **9.6억 → 1억 수준**

### 개발기간 단축
- 예시: **1년 → 1~2달 수준**

### 운영 유연성
- 단말이 어디에 있든 인터넷 연결만 되면 제어 가능
- 국가/거점이 달라도 동일 방식 적용 가능
- 예시: **6개국 32명 운영 환경**에도 확장 가능

## As-is / To-be 비교 포인트
| 구분 | As-is | To-be |
|---|---|---|
| 테스트 중심 | 서버 중심 | 단말 중심 |
| 절차 수 | 많음 | 적음 |
| 사람 개입 | 큼 | 작음 |
| 구축 복잡도 | 높음 | 낮음 |
| 확장성 | 제한적 | 높음 |

## 발표 스크립트
기대효과는 크게 세 가지입니다.  
첫째, **테스트가 단순해집니다.** 지금은 여러 시스템과 절차가 필요하지만, 앞으로는 단말 안에서 대부분 처리할 수 있기 때문에 흐름이 짧아집니다.  
둘째, **비용을 줄일 수 있습니다.** 서버, 연결, 저장, 판독, 업로드가 각각 따로 움직이던 구조가 줄어들기 때문에 개발비와 운영비를 함께 줄일 수 있습니다.  
셋째, **운영이 유연해집니다.** 단말이 어디에 있든 연결만 되어 있으면 제어가 가능하기 때문에, 국내뿐 아니라 해외 거점 운영에도 유리합니다.  
한마디로 정리하면, **더 단순하게, 더 싸게, 더 넓게 운영할 수 있는 구조**입니다.

# [슬라이드] #4 (3분)
## 제목
**4. 데모 영상 시연**

## 구성
- To-be 시나리오 기반 영상 시연
- 흐름 중심으로 보여주기
  - 사용자가 자연어로 검증 요청
  - 단말 안의 AI가 앱 실행/검증 수행
  - 결과 저장 및 판정
  - 결과 업로드까지 이어지는 모습 제시

## 발표 스크립트
마지막으로 실제 데모 영상을 보시겠습니다.  
이 영상은 To-be 시나리오를 기준으로 준비했습니다.  
보실 포인트는 아주 단순합니다. **사람이 손으로 하던 일을, 단말 안의 AI가 대신 처리하는가**입니다.  
즉, 명령을 이해하고, 실제로 움직이고, 결과를 남기고, 최종 판정까지 이어지는 흐름을 보시면 됩니다.  
이 부분이 확인되면, 오늘 말씀드린 단순화와 비용 절감 가능성을 직관적으로 보실 수 있습니다.
