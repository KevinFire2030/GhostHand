# 브라우징 덤프 방법

이 문서는 사용자가 브라우저에서 직접 수행한 클릭, 입력, 파일 업로드, 페이지 이동 등의 행동을 덤프하고, 그 덤프를 Codex가 참조하여 브라우징 자동화 스크립트로 재현하기 위한 방법을 정리한다.

목표는 특정 사이트 전용 자동화가 아니라, Gmail, 업무 포털, 관리자 페이지, SaaS 대시보드 등 다양한 웹 브라우징 작업에 재사용할 수 있는 범용 기록 방식을 만드는 것이다.

## 핵심 아이디어

브라우징 덤프는 단순 화면 녹화가 아니라, 자동화에 필요한 정보를 구조화해서 남기는 것이다.

기록해야 할 정보는 다음과 같다.

- 어떤 URL에 접속했는지
- 어떤 브라우저 컨텍스트를 사용했는지
- 어떤 요소를 클릭했는지
- 어떤 값을 입력했는지
- 어떤 키보드 조작을 했는지
- 어떤 파일을 업로드했는지
- 어떤 대기 조건이 필요했는지
- 로그인, 2FA, CAPTCHA처럼 자동화가 어려운 구간이 있었는지
- 최종 성공 여부를 어떻게 검증할 수 있는지

이 정보를 남겨두면 Codex는 덤프를 읽고 Playwright 같은 브라우징 자동화 도구로 재현 스크립트를 만들 수 있다.

## 권장 도구

### 1. Playwright

가장 권장하는 도구다. 브라우저 자동화, 액션 기록, 셀렉터 추출, 파일 업로드, 네트워크 대기, 스크린샷 저장을 모두 지원한다.

주요 용도:

- 사용자의 행동을 코드로 기록
- 페이지 클릭과 입력 자동화
- 파일 업로드 자동화
- 로그인 세션 재사용
- 실행 중 스크린샷과 trace 저장
- Codex가 읽을 수 있는 TypeScript 또는 JavaScript 스크립트 생성

권장 명령:

```bash
npx playwright codegen --save-storage=auth-state.json https://example.com
```

로그인 상태를 저장한 뒤 재사용할 때:

```bash
npx playwright codegen --load-storage=auth-state.json https://example.com
```

자동화 실행 시 trace를 남길 때:

```bash
npx playwright test --trace on
```

### 2. Chrome DevTools Recorder

Chrome에 내장된 Recorder도 사용할 수 있다.

주요 용도:

- 사용자가 브라우저에서 수행한 행동을 기록
- Puppeteer 또는 Playwright 형식으로 export
- 클릭 대상과 입력 흐름을 빠르게 확인

장점은 별도 설치 없이 Chrome에서 바로 기록할 수 있다는 점이다. 단, 장기적으로 Codex가 자동화 스크립트를 만들기에는 Playwright codegen과 trace가 더 안정적이다.

### 3. Playwright Trace Viewer

자동화가 실패했을 때 원인을 분석하는 데 좋다.

사용 방법:

```bash
npx playwright show-trace trace.zip
```

Trace에는 각 단계별 DOM 상태, 스크린샷, 네트워크 요청, 콘솔 로그가 포함된다.

### 4. HAR 파일

네트워크 요청 흐름이 중요한 사이트에서는 HAR 파일도 함께 저장한다.

주요 용도:

- 로그인 후 어떤 API가 호출되는지 확인
- 폼 제출 후 어떤 요청이 나가는지 확인
- 파일 업로드 요청 구조 확인

다만 HAR에는 민감한 쿠키, 토큰, 헤더가 포함될 수 있으므로 공유 전에 반드시 비식별화해야 한다.

## 덤프 산출물 구성

브라우징 덤프는 다음 파일 묶음으로 남기는 것을 권장한다.

```text
browsing-dump/
  README.md
  steps.yaml
  recording.spec.ts
  auth-state.json
  trace.zip
  screenshots/
    001-open-page.png
    002-login-form.png
    003-compose.png
  uploads/
    sample.pdf
```

각 파일의 역할은 다음과 같다.

- `README.md`: 사람이 읽는 설명
- `steps.yaml`: 클릭, 입력, 업로드, 대기 조건을 구조화한 원본 덤프
- `recording.spec.ts`: Playwright codegen 또는 수동 정리된 재현 코드
- `auth-state.json`: 로그인 세션 저장 파일
- `trace.zip`: 실행 trace
- `screenshots/`: 주요 단계 화면
- `uploads/`: 자동화 테스트에 사용할 샘플 업로드 파일

민감한 실제 파일은 `uploads/`에 넣지 말고 샘플 파일로 대체한다.

## 기록해야 하는 항목

각 사용자 행동은 가능한 한 다음 필드를 포함해서 기록한다.

```yaml
- step: 1
  action: goto
  url: "https://example.com"
  wait_for: "networkidle"
  screenshot: "screenshots/001-open-page.png"

- step: 2
  action: click
  target:
    role: "button"
    name: "로그인"
    selector: "button:has-text('로그인')"
  note: "상단 오른쪽 로그인 버튼"

- step: 3
  action: fill
  target:
    label: "이메일"
    selector: "input[type='email']"
  value_type: "credential.email"
  value_example: "user@example.com"

- step: 4
  action: upload
  target:
    selector: "input[type='file']"
  file:
    original_path: "C:/Users/me/Documents/report.pdf"
    dump_path: "uploads/sample.pdf"
```

### 필수 필드

- `step`: 순서
- `action`: 행동 종류
- `target`: 대상 요소
- `value_type`: 실제 값 대신 값의 의미
- `wait_for`: 다음 단계로 넘어가기 전 대기 조건
- `screenshot`: 해당 단계의 화면 증거
- `note`: 사람이 이해하기 위한 설명

### action 종류

자주 사용하는 action은 다음과 같다.

- `goto`: URL 접속
- `click`: 클릭
- `dblclick`: 더블 클릭
- `fill`: 입력 필드 채우기
- `press`: 키보드 입력
- `select`: 드롭다운 선택
- `check`: 체크박스 선택
- `uncheck`: 체크박스 해제
- `upload`: 파일 업로드
- `wait`: 대기
- `assert`: 성공 여부 검증
- `manual`: 자동화가 어려워 사람이 직접 처리해야 하는 단계

## 셀렉터 기록 원칙

자동화 스크립트는 셀렉터 품질에 크게 영향을 받는다.

권장 우선순위:

1. role 기반 셀렉터
2. label 기반 셀렉터
3. placeholder 기반 셀렉터
4. test id 기반 셀렉터
5. 안정적인 CSS 셀렉터
6. XPath 또는 좌표 기반 클릭

Playwright 예시:

```ts
await page.getByRole('button', { name: '로그인' }).click();
await page.getByLabel('이메일').fill(process.env.USER_EMAIL!);
await page.getByPlaceholder('검색어 입력').fill('invoice');
await page.getByTestId('submit-button').click();
```

좌표 기반 클릭은 마지막 수단으로만 사용한다. 화면 크기, 언어, 레이아웃 변경에 취약하기 때문이다.

## 입력값 기록 원칙

실제 비밀번호, API 키, 주민등록번호, 결제 정보, 개인 문서 경로는 덤프에 직접 저장하지 않는다.

대신 다음처럼 값의 의미만 남긴다.

```yaml
value_type: "credential.password"
value_env: "GMAIL_PASSWORD"
value_example: "********"
```

자동화 스크립트에서는 환경 변수를 사용한다.

```ts
await page.getByLabel('비밀번호').fill(process.env.GMAIL_PASSWORD!);
```

## 파일 업로드 기록 원칙

파일 업로드는 반드시 다음 정보를 남긴다.

- 파일 입력 요소의 셀렉터
- 사용자가 선택한 원본 파일 경로
- 덤프에 포함할 샘플 파일 경로
- 파일 형식
- 업로드 완료를 확인하는 조건

예시:

```yaml
- step: 8
  action: upload
  target:
    selector: "input[type='file']"
  file:
    original_path: "C:/Users/me/Desktop/report.pdf"
    dump_path: "uploads/sample-report.pdf"
    mime_type: "application/pdf"
  wait_for:
    text: "첨부 완료"
```

Playwright 예시:

```ts
await page.locator('input[type="file"]').setInputFiles('uploads/sample-report.pdf');
await expect(page.getByText('첨부 완료')).toBeVisible();
```

## 로그인 처리

로그인은 자동화가 흔히 실패하는 구간이다. 가능하면 사용자가 한 번 로그인하고 세션 상태를 저장한다.

```bash
npx playwright codegen --save-storage=auth-state.json https://mail.google.com
```

이후 자동화에서는 저장된 세션을 사용한다.

```ts
const context = await browser.newContext({
  storageState: 'auth-state.json',
});
```

2FA, CAPTCHA, 보안 경고가 있는 경우에는 덤프에 `manual` 단계로 표시한다.

```yaml
- step: 4
  action: manual
  reason: "2FA 인증 앱 승인 필요"
  resume_when: "받은편지함이 보이면 계속"
```

## Gmail 예시

Gmail은 예시일 뿐이다. 이 구조는 다른 모든 웹사이트에도 동일하게 적용한다.

사용자 행동:

1. Chrome으로 Gmail 접속
2. 사용자 로그인
3. 메일 작성
4. 제목 입력
5. 받는 사람 입력
6. 내용 입력
7. 첨부파일 선택
8. 보내기 클릭

구조화 덤프 예시:

```yaml
task: "send_gmail_with_attachment"
site: "Gmail"
start_url: "https://mail.google.com"
browser: "chromium"
storage_state: "auth-state.json"

steps:
  - step: 1
    action: goto
    url: "https://mail.google.com"
    wait_for:
      role: "button"
      name: "편지쓰기"
    screenshot: "screenshots/001-inbox.png"

  - step: 2
    action: click
    target:
      role: "button"
      name: "편지쓰기"
    screenshot: "screenshots/002-compose-open.png"

  - step: 3
    action: fill
    target:
      selector: "input[aria-label='받는사람']"
    value_type: "email.to"
    value_env: "MAIL_TO"

  - step: 4
    action: fill
    target:
      selector: "input[name='subjectbox']"
    value_type: "email.subject"
    value_env: "MAIL_SUBJECT"

  - step: 5
    action: fill
    target:
      role: "textbox"
      name: "메일 본문"
    value_type: "email.body"
    value_env: "MAIL_BODY"

  - step: 6
    action: upload
    target:
      selector: "input[type='file']"
    file:
      dump_path: "uploads/sample.pdf"
    wait_for:
      text: "sample.pdf"

  - step: 7
    action: click
    target:
      role: "button"
      name: "보내기"
    wait_for:
      text: "메일을 보냈습니다"

  - step: 8
    action: assert
    expect:
      text: "메일을 보냈습니다"
```

Playwright 스크립트 예시:

```ts
import { chromium, expect } from '@playwright/test';

async function main() {
  const browser = await chromium.launch({ headless: false });
  const context = await browser.newContext({
    storageState: 'auth-state.json',
  });
  const page = await context.newPage();

  await page.goto('https://mail.google.com');
  await page.getByRole('button', { name: '편지쓰기' }).click();

  await page.locator("input[aria-label='받는사람']").fill(process.env.MAIL_TO!);
  await page.locator("input[name='subjectbox']").fill(process.env.MAIL_SUBJECT!);
  await page.getByRole('textbox', { name: '메일 본문' }).fill(process.env.MAIL_BODY!);

  await page.locator("input[type='file']").setInputFiles('uploads/sample.pdf');
  await expect(page.getByText('sample.pdf')).toBeVisible();

  await page.getByRole('button', { name: '보내기' }).click();
  await expect(page.getByText('메일을 보냈습니다')).toBeVisible();

  await browser.close();
}

main();
```

## Codex가 덤프를 사용할 때의 절차

Codex는 덤프를 받으면 다음 순서로 자동화 스크립트를 만든다.

1. `README.md`와 `steps.yaml`을 읽고 작업 목표를 이해한다.
2. 로그인 상태가 필요한지 확인한다.
3. `auth-state.json`이 있으면 재사용한다.
4. `recording.spec.ts`가 있으면 먼저 실행 가능한지 확인한다.
5. 불안정한 셀렉터를 role, label, test id 기반으로 정리한다.
6. 실제 값은 환경 변수로 분리한다.
7. 파일 업로드는 샘플 파일로 검증한다.
8. 성공 조건을 `expect`로 명시한다.
9. 실패 시 trace와 screenshot을 저장한다.

## 스킬로 만들 때 포함할 기능

최종적으로 Codex 스킬로 배포하려면 다음 기능을 포함하는 것이 좋다.

- 새 브라우징 덤프 폴더 생성
- Playwright codegen 실행 안내
- `steps.yaml` 템플릿 생성
- 업로드 파일 샘플화
- `auth-state.json` 저장 및 재사용 안내
- trace와 screenshot 저장
- 덤프를 읽고 Playwright 자동화 스크립트 생성
- 실행 실패 시 trace 분석
- 민감 정보 검사
- 스크립트 재실행 검증

스킬의 기본 명령 흐름은 다음과 같다.

```text
1. 사용자에게 작업 이름과 시작 URL을 받는다.
2. Playwright codegen으로 사용자의 실제 행동을 기록한다.
3. 기록된 코드를 steps.yaml로 정리한다.
4. 민감 정보를 환경 변수로 치환한다.
5. 업로드 파일은 샘플 파일로 대체한다.
6. Codex가 재현 가능한 Playwright 스크립트를 생성한다.
7. 스크립트를 실행하고 trace와 screenshot으로 검증한다.
8. 실패하면 셀렉터와 대기 조건을 보정한다.
```

## 주의사항

- 비밀번호, 쿠키, 세션 토큰, 개인 파일, 결제 정보는 그대로 커밋하지 않는다.
- `auth-state.json`은 민감 정보로 취급하고 공개 저장소에 올리지 않는다.
- CAPTCHA 우회나 서비스 약관을 위반하는 자동화는 만들지 않는다.
- 이메일 발송, 결제, 삭제, 배포처럼 실제 영향을 주는 작업은 dry-run 또는 확인 단계를 둔다.
- 실제 운영 데이터 대신 샘플 데이터를 사용해 자동화를 검증한다.

## 결론

브라우징 덤프의 핵심은 사용자의 행동을 사람이 보는 영상이 아니라 기계가 재현 가능한 절차로 남기는 것이다. Playwright codegen, trace, screenshot, `steps.yaml`을 함께 사용하면 Codex가 덤프를 읽고 안정적인 브라우징 자동화 스크립트를 만들 수 있다.

이 방식은 Gmail 같은 예시뿐 아니라 로그인, 폼 작성, 파일 업로드, 관리자 페이지 조작, SaaS 반복 업무 자동화에 범용적으로 적용할 수 있다.
