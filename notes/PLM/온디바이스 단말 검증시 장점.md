# 온디바이스 단말 검증시 장점

스마트폰 기능 검증 테스트에서 1차 로그 분석을 온디바이스에서 수행할 경우의 장점을 정리합니다.

## 결론

- 온디바이스에서 1차 분석까지 되면 담당자 지정이 훨씬 쉬워짐
- 단순 편의 수준이 아니라 분류 정확도, 전달 속도, triage 효율이 크게 올라감
- 실패 직후 바로 유형 분류와 담당자 후보 추천이 가능해짐

## 왜 담당자 지정이 쉬워지는가

온디바이스 1차 분석 결과에서 아래 신호를 빠르게 뽑을 수 있습니다.

- crash 키워드
- ANR 여부
- 앱 패키지
- 발생 화면
- 기능 영역
- 네트워크/API 오류
- 권한 문제
- 카메라/오디오/센서 등 HW 관련 로그
- 테스트 스크립트 실패 여부

이 신호들을 바탕으로 아래 구조로 연결할 수 있습니다.

- 로그 분석 결과
- 문제 카테고리 분류
- 담당 조직/담당자 추천

## 담당자 지정 예시

예를 들면 아래와 같은 자동 매핑이 가능합니다.

- `CameraService`, `camera provider`, preview 실패
  - 카메라 담당
- `Permission Denial`, `SecurityException`
  - 플랫폼/권한 담당
- `SocketTimeoutException`, `HTTP 500`, API fail
  - 서버/API 또는 연동 담당
- 앱 패키지 내부 `NullPointerException`
  - 앱 개발 담당
- `selector mismatch`, `uiautomator timeout`
  - 테스트 자동화 담당

즉,

`로그 분석 결과 → 문제 카테고리 → 담당 조직/담당자`

구조를 만들 수 있습니다.

## 추천 분류 구조

### 1단계: 실패 유형 분류

예:
- `CRASH`
- `ANR`
- `NETWORK`
- `PERMISSION`
- `UI`
- `TEST_SCRIPT`
- `DEVICE_ENV`

### 2단계: 기능 영역 분류

예:
- Camera
- Audio
- Display
- Connectivity
- Account/Login
- Settings
- Third-party App
- Automation Infra

### 3단계: 담당자 매핑

예:
- `CRASH + Camera` → 카메라 개발 담당
- `NETWORK + Login` → 계정/API 담당
- `TEST_SCRIPT + UI selector` → 자동화 담당
- `DEVICE_ENV + permission` → 플랫폼 담당

## 실무에서 특히 좋은 점

### 1) triage 속도 향상

기존 방식:
- 사람이 로그를 열어봄
- 누가 볼지 고민함
- 잘못 전달되면 다시 재분배됨

온디바이스 분석 후:
- 실패 직후 자동 분류
- 담당자 추천 생성
- 바로 이슈 초안 작성 가능

### 2) 이슈 품질 향상

이슈에 아래 정보를 자동으로 넣을 수 있습니다.

- 실패 유형
- 추정 원인
- 추천 담당자
- 관련 로그 키워드
- 관련 모듈

받는 사람 입장에서 이해 속도가 빨라집니다.

### 3) 담당자 자동화 가능

예:
- GitHub Issue assignee 자동 지정
- label 자동 지정
- PLM 등록 시 `inChargeUser` 후보 자동 추천

## 온디바이스에서 가능한 수준

### 쉬운 것
- 키워드 기반 분류
- 앱/모듈/기능영역 분류
- 담당자 룰 매핑

### 조금 더 고급
- 시나리오 맥락 반영
- 이전 유사 오류와 비교
- 재현 빈도 기반 우선순위 추천

### 어려운 것
- 원인을 100% 정확히 특정
- 복합 장애를 완벽히 한 사람에게만 매핑

따라서 추천은 아래와 같습니다.

- 온디바이스는 1차 추천
- 최종 확정은 서버나 사람이 검토

## 가장 좋은 운영 방식

### 온디바이스 역할
- 로그 수집
- 1차 분류
- 담당자 후보 추천
- 이슈/PLM 초안 생성

### 서버/iPaaS/백엔드 역할
- 유사 이슈 비교
- 조직/담당자 최신 매핑 반영
- 최종 할당
- 등록/배포

이렇게 역할을 나누면 가장 안정적입니다.

## 예시 출력

```json
{
  "scenarioId": "UT-004",
  "result": "FAIL",
  "category": "NETWORK",
  "module": "Login/API",
  "rootCauseHint": "HTTP 500 and auth timeout detected",
  "recommendedOwner": "api-owner",
  "recommendedTeam": "Account Backend",
  "confidence": 0.82
}
```

또는:

```json
{
  "scenarioId": "UT-007",
  "result": "FAIL",
  "category": "TEST_SCRIPT",
  "module": "Automation Infra",
  "rootCauseHint": "selector mismatch and timeout waiting for object",
  "recommendedOwner": "qa-automation",
  "recommendedTeam": "QA Automation",
  "confidence": 0.91
}
```

이런 형식이면 자동 assign이 훨씬 쉬워집니다.

## 이 구조가 특히 좋은 이유

다음 흐름과 자연스럽게 연결됩니다.

1. 실패 감지
2. 로그 분석
3. 문제 유형 분류
4. 담당자 추천
5. GitHub Issue assignee 자동 지정
6. PLM `inChargeUser` 후보 자동 입력

즉, 온디바이스 분석 결과가 GitHub와 PLM 양쪽 모두에 활용될 수 있습니다.

## 한 줄 정리

- 1차 분석을 온디바이스에서 하면 담당자 지정을 훨씬 쉽게 할 수 있음
- 핵심 구조는 `문제 유형 분류 → 기능 영역 분류 → 담당자 매핑`
- 다만 온디바이스는 1차 추천, 최종 확정은 중앙 시스템 또는 사람 검토가 가장 안정적임
