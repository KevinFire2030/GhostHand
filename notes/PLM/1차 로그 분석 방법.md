# 1차 로그 분석 방법

스마트폰 기능 검증 테스트에서 시나리오별로 저장한 로그를 바탕으로, 문제점 등록 전에 1차 로그 분석을 수행하는 방법을 정리합니다.

## 결론

- 저장한 로그로 문제점 등록 전에 1차 로그 분석이 충분히 가능함
- 실무적으로도 `로그 저장 → 1차 분석 → 이슈/문제점 등록` 순서가 더 좋음
- 이 과정을 거치면 중복 이슈를 줄이고, 문제 등록 품질을 높일 수 있음

## 왜 1차 로그 분석이 필요한가

1차 로그 분석을 먼저 하면 아래 이점이 있습니다.

- 중복 이슈 감소
- 단순 환경 문제 선별 가능
- 문제점 제목/내용 정확도 향상
- GitHub Issue / PLM 등록 품질 향상
- 테스트 스크립트 문제와 실제 단말/앱 문제 구분 가능

## 1차 로그 분석에서 할 수 있는 것

### 빠르게 거를 수 있는 항목

- 앱 crash 여부
- ANR 여부
- permission 거부
- 네트워크 오류
- 타임아웃
- 특정 API 실패
- 테스트 스크립트 문제 여부
- 단말/빌드/환경 문제 여부

### 자동 분류 예시

실패 유형을 아래처럼 자동 분류할 수 있습니다.

- `CRASH`
- `ANR`
- `UI mismatch`
- `Network`
- `Permission`
- `Test script issue`
- `Unknown`

이렇게 1차 분류한 뒤 등록하면 관리가 훨씬 깔끔해집니다.

## 추천 운영 흐름

### 기본 흐름

1. 시나리오 실행
2. 영상/로그 저장
3. 1차 로그 분석
4. 결과 분류
5. 필요 시 GitHub Issue 등록
6. 정말 필요한 건만 PLM 등록

이 흐름의 장점은 “무조건 등록”이 아니라 “분석 후 등록”이라 노이즈가 줄어든다는 점입니다.

## 1차 분석 결과로 뽑으면 좋은 항목

추천 출력 항목은 아래와 같습니다.

- 시나리오명
- 결과: PASS / FAIL
- 실패 유형
- 핵심 에러 키워드
- 추정 원인
- 관련 로그 구간
- 문제 등록 필요 여부
- 추천 제목 초안
- 재현 경로 초안

### 예시 JSON

```json
{
  "scenarioId": "UT-002",
  "result": "FAIL",
  "category": "CRASH",
  "rootCauseHint": "Camera service timeout 후 FATAL EXCEPTION 발생",
  "keyLogs": [
    "FATAL EXCEPTION: main",
    "Camera service timeout"
  ],
  "needsIssue": true,
  "issueTitle": "[UT][FAIL] Camera launch crashes after app start",
  "reproduction": [
    "카메라 앱 실행",
    "권한 허용",
    "preview 확인 시도",
    "앱 비정상 종료"
  ]
}
```

이렇게 나오면 GitHub Issue 초안이나 PLM defect 초안으로 바로 이어질 수 있습니다.

## 분석 수준 3단계

### 1. 규칙 기반

가장 먼저 추천하는 방식입니다.

예:
- `FATAL EXCEPTION` 있으면 crash
- `ANR in` 있으면 ANR
- `Permission Denial` 있으면 권한 문제
- `timeout` 많으면 응답 지연/네트워크 의심

장점:
- 빠름
- 구현 쉬움
- 실무 적용이 쉬움

### 2. 규칙 + 메타데이터

아래 정보를 함께 보면 더 정확해집니다.

- 로그
- 시나리오 이름
- 앱 패키지
- 테스트 단계
- 단말 정보

예:
- 카메라 시나리오 + camera 관련 에러
- 네트워크 시나리오 + socket timeout
- 로그인 시나리오 + auth fail

### 3. LLM 기반 요약/판단

규칙 기반 결과를 LLM이 읽고 아래 작업을 수행할 수 있습니다.

- 요약
- 사람 친화적 설명
- 이슈 제목/본문 생성
- PLM defect 초안 생성

추천 구조는 아래와 같습니다.

- 1차 판정은 규칙 기반
- 최종 문서화는 LLM

## 실무에서 특히 좋은 포인트

### 1) 등록 필요 여부 판단

모든 실패를 다 올리면 노이즈가 많아집니다.

1차 분석에서 아래 항목은 분리할 수 있습니다.

- 재현성 낮음
- 환경 문제
- 이미 알려진 오류
- 테스트 스크립트 오동작

예:

- `needsIssue = false`
- `reason = flaky test suspected`

### 2) 중복 감지

기존 GitHub Issue나 기존 PLM defect와 비교해 아래 기준으로 중복 의심 여부를 태깅할 수 있습니다.

- 비슷한 에러 키워드
- 같은 시나리오
- 같은 빌드
- 같은 failure signature

### 3) 우선순위 보조

예:
- crash + 재현율 높음 → High
- UI 문구 불일치 → Medium
- 일회성 timeout → Low 또는 pending

## 추천 자동화 구조

### 시나리오 종료 후 자동 실행

예를 들어 아래와 같은 구조를 권장합니다.

- `analyze_log.py`

입력:
- `logcat.txt`
- `meta.json`
- 필요 시 `result.json`

출력:
- `analysis.json`
- `issue.md`
- `plm-draft.md`

### 분기 처리

- `needsIssue = true` 이면 GitHub Issue 등록
- `needsPlm = true` 이면 PLM defect 초안 생성
- 아니면 로컬 기록만 남김

## 1차 분석에서 볼 키워드 예시

### Crash / Runtime

- `FATAL EXCEPTION`
- `AndroidRuntime`
- `Process ... has died`
- `SIGSEGV`
- `abort`

### ANR / Hang

- `ANR in`
- `Input dispatching timed out`
- `Broadcast of Intent ... timed out`

### Permission / Security

- `Permission Denial`
- `SecurityException`

### Network

- `SocketTimeoutException`
- `UnknownHostException`
- `ConnectException`
- `HTTP 500`
- `HTTP 403`

### App logic / UI

- `IllegalStateException`
- `NullPointerException`
- `IndexOutOfBoundsException`

### Test script issue

- 요소 못 찾음
- timeout waiting for object
- adb input 실패
- selector mismatch

## 영상과 같이 보는 것이 좋은 이유

로그만 봐도 1차 분석은 가능하지만, 영상/스크린샷과 함께 보면 정확도가 더 높아집니다.

예:
- 로그에는 crash가 없음
- 영상상 UI가 멈춤
- 이 경우 ANR / 프리징 / 테스트 로직 문제를 추가 의심할 수 있음

즉,

- `로그 = 내부 증거`
- `영상 = 외부 증거`

둘을 같이 보면 정확도가 크게 올라갑니다.

## 추천 운영 정책

### GitHub Issue 등록 전

자동으로 아래를 만들면 좋습니다.

- 실패 유형
- 핵심 로그 5~20줄
- 시나리오 정보
- 영상 경로
- 단말 정보
- 제목 초안
- 재현 단계 초안

### PLM 등록 전

한 번 더 정제해서 아래 항목으로 변환합니다.

- 외부 시스템 문제점 ID
- 제목
- Content
- 재현경로
- 예상결과
- 중요도
- 발생빈도
- 첨부 여부

즉,

- GitHub는 개발 협업용
- PLM은 공식 문제 등록용

으로 역할을 나눌 수 있습니다.

## 한 줄 정리

- 저장한 로그로 문제점 등록 전에 1차 로그 분석이 충분히 가능함
- 오히려 가장 좋은 운영 방식은 `영상/로그 저장 → 1차 분석 → GitHub Issue → 필요 시 PLM 등록`
- 추천 구조는 `규칙 기반 판정 + LLM 문서화` 조합임
