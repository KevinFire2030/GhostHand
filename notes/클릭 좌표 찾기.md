# 클릭 좌표 찾기

## 개요
무선 ADB로 앱을 실행하고 검색창, 버튼, 탭 같은 UI 요소를 클릭할 때 위치를 찾는 방법을 정리한다.
실전에서는 아래 3가지 방식을 조합해서 사용한다.

1. 고정 좌표로 클릭
2. 화면 UI 구조(XML) 읽어서 위치 찾기
3. 스크린샷 보고 좌표 추정하기

가장 안정적인 방법은 **UI XML을 읽어서 bounds 기준으로 좌표를 계산하는 방식**이다.

---

## 1. 고정 좌표로 클릭
가장 단순한 방식이다.

```bash
adb shell input tap 540 180
```

의미:
- 화면의 `(x=540, y=180)` 위치를 터치한다.

장점:
- 빠르다.
- 바로 테스트 가능하다.

단점:
- 해상도나 DPI가 바뀌면 어긋날 수 있다.
- 앱 레이아웃이 바뀌면 잘못 눌릴 수 있다.
- 추천 카드, 광고, 탭 위치 변화에 취약하다.

---

## 2. UI XML 덤프로 위치 찾기
가장 추천하는 방식이다.
현재 화면의 UI 트리를 뽑아서 버튼/입력창의 bounds를 찾는다.

### 2-1. UI 덤프 생성
```bash
adb shell uiautomator dump /sdcard/ui.xml
adb pull /sdcard/ui.xml .
```

덤프 결과 예시:
```xml
<node text="" content-desc="Google Play 검색" bounds="[124,151][808,219]" />
```

여기서 확인 가능한 정보:
- text
- content-desc
- class
- bounds

### 2-2. 중심 좌표 계산
예를 들어 bounds가 아래와 같으면:

- `[124,151][808,219]`

중심 좌표는:
- x = (124 + 808) / 2 = 466
- y = (151 + 219) / 2 = 185

그래서 실제 클릭은:

```bash
adb shell input tap 466 185
```

### 2-3. 장점
- 요소 이름(`검색`, `설치`, `Instagram`) 기준으로 찾을 수 있다.
- 감으로 누르는 것보다 훨씬 정확하다.
- 반복 자동화에 적합하다.

### 2-4. 단점
- 일부 앱은 UI 트리에 정보가 부족하다.
- WebView/커스텀 UI/특수 레이아웃에서는 텍스트가 안 잡힐 수 있다.

---

## 3. 스크린샷으로 좌표 추정
UI XML이 잘 안 잡힐 때 사용하는 방식이다.

```bash
adb exec-out screencap -p > screen.png
```

스크린샷을 보고:
- 버튼이 상단인지 하단인지
- 좌측인지 우측인지
- 대략 어느 위치인지
판단해서 좌표를 잡는다.

예:
- 상단 오른쪽 돋보기 → `985 180`
- 하단 검색 탭 → `540 2260`

장점:
- UI XML이 안 잡혀도 쓸 수 있다.

단점:
- 사람이 감으로 잡기 때문에 오차가 날 수 있다.
- 추천/광고/정렬이 바뀌면 다른 항목이 눌릴 수 있다.

---

## 4. 앱 실행 방식
앱 실행은 좌표 클릭보다 패키지명/액티비티 기반으로 하는 게 더 안정적이다.

### 4-1. monkey 사용
```bash
adb shell monkey -p com.android.vending -c android.intent.category.LAUNCHER 1
```

예:
- 구글 플레이 실행

### 4-2. activity 직접 실행
```bash
adb shell am start -n 패키지명/액티비티명
```

예:
```bash
adb shell cmd activity start-activity \
  -a android.intent.action.MAIN \
  -c android.intent.category.LAUNCHER \
  -n com.samsung.android.themestore/.activity.LauncherFromFinder
```

예:
- 삼성 테마스토어 실행

정리:
- 앱 실행 → 패키지/액티비티 기반
- 화면 조작 → UI 분석 또는 좌표 기반

---

## 5. 검색어 입력 방법
영문 입력은 비교적 안정적이다.

```bash
adb shell input text instagram
```

검색 실행:

```bash
adb shell input keyevent KEYCODE_ENTER
```

### 한글 입력 주의
```bash
adb shell input text 무료
```

이 방식은 한글 입력이 잘 안 될 수 있다.

대응 방법:
- 사용자가 직접 입력
- ADB 키보드(IME) 설치
- 영문 대체어 사용 (`free` 등)

---

## 6. 실전 작업 순서
실무에서는 보통 아래 순서로 작업한다.

### Step 1. 앱 실행
```bash
adb shell monkey -p com.android.vending -c android.intent.category.LAUNCHER 1
```

### Step 2. 현재 화면 UI 덤프
```bash
adb shell uiautomator dump /sdcard/ui.xml
adb pull /sdcard/ui.xml .
```

### Step 3. 찾고 싶은 요소 검색
예:
- 검색
- 설치
- Instagram
- 로그인

### Step 4. bounds 기준 중심 좌표 계산
예:
- `[124,151][808,219]` → `(466,185)`

### Step 5. tap/input 실행
```bash
adb shell input tap 466 185
adb shell input text instagram
adb shell input keyevent KEYCODE_ENTER
```

### Step 6. 결과 검증
```bash
adb exec-out screencap -p > result.png
```

---

## 7. 왜 잘못 눌릴 수 있는가
아래 이유로 자동화가 틀어질 수 있다.

- UI XML에 텍스트가 충분히 안 잡힘
- 같은 `검색` 요소가 여러 개 존재함
- 광고/추천 카드 순서가 바뀜
- 앱이 WebView 기반이라 UI 정보가 빈약함
- 화면 전환 속도가 느려서 타이밍이 밀림
- 키보드가 떠서 실제 좌표가 달라짐
- 하단 탭/상단 탭이 상태에 따라 변함

---

## 8. 안정적으로 하려면
추천 방식:

### 빠른 방식
- `uiautomator dump` + `input tap`

### 더 안정적인 방식
- `uiautomator dump`
- 텍스트 매칭
- bounds 중심 좌표 계산
- 클릭 후 스크린샷 검증

### 한글까지 자동화하고 싶다면
- ADB Keyboard 같은 입력 IME 세팅 필요

---

## 9. 핵심 요약
클릭 좌표는 보통 아래 순서로 찾는다.

1. 앱 실행
2. `uiautomator dump`로 UI XML 추출
3. `text` 또는 `content-desc`에서 원하는 요소 탐색
4. `bounds` 중심 좌표 계산
5. `adb shell input tap x y` 실행
6. 스크린샷으로 결과 확인

UI 정보가 부족하면 스크린샷 보고 좌표를 보정한다.

---


```bash

```

```bash
adb shell input tap 466 185
```

```bash
adb shell input text instagram
```

```bash
adb shell input keyevent KEYCODE_ENTER
```

```bash
adb shell input keyevent KEYCODE_HOME
```

### 뒤로 가기
```bash
adb shell input keyevent KEYCODE_BACK
```
